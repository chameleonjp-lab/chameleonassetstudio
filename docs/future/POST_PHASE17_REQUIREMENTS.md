## 1. 目的

この文書は、Phase 17 完了後に Chameleon Asset Studio をさらに伸ばすための要件を定義する。

ここで扱うのは、ローカル中心のゲームアセット制作ツールとしての拡張である。販売、課金、クラウド、チーム利用、SaaS 運用は扱わない。

この文書の目的は、今後 `claude-fable-5` が使えない場合でも、仕様の抜け漏れや誤実装を避けることである。

---

## 2. 前提

Phase 17 完了時点で、最低限次が満たされていることを前提にする。

- 2D アセットを作成できる。
- 画像を取り込み、編集できる。
- レイヤー、パーツ、原点、アンカー、当たり判定を設定できる。
- フレームアニメーションを作成できる。
- 簡易リグをフレームへ焼き込める。
- キャラクター、アイテム、背景、タイル、ギミック、エフェクトを最低限扱える。
- PNG / WebP / JSON / Sprite Sheet / Atlas / ZIP を書き出せる。
- Canvas / PixiJS / Phaser の examples を出せる。
- helper snippet と Godot / Unity 取り込みガイドを出せる。
- `.casproj` を読み書きできる。

---

## 3. 明示的に対象外とするもの

以下は、別フェーズで扱う。Post Phase 17 のローカル制作要件には含めない。

- ユーザーアカウント
- クラウド保存
- チーム編集
- 課金
- 請求書払い
- SLA
- 企業向け導入支援
- 本格的な AI 生成サーバー運用
- 既存ゲームプロジェクトへの双方向同期
- 本格的な Unity / Godot importer
- CI 用 CLI
- SaaS セキュリティ監査

これらを入れない理由は、不要だからではなく、ローカル制作体験を固める前に入れると、設計が散らばるためである。

## 4. 2D 制作体験の不足を埋める要件

### 4.1 画像編集の細かさ

現在の画像編集は、ゲーム用に最低限整えるための機能である。将来は次を検討する。

| 機能 | 目的 | 優先度 |
|---|---|---|
| ブラシ | 手描き補正、透明化、色追加 | 中 |
| 塗りつぶし | アイテムやキャラの色替え | 中 |
| 選択範囲 | 部分編集 | 中 |
| 反転 | 左右差分、向き変更 | 高 |
| 整列 | パーツ位置合わせ | 高 |
| スナップ | 原点、アンカー、判定の正確化 | 高 |
| グリッド | ドット絵、タイル編集 | 高 |
| パレット管理 | 色違い量産 | 中 |
| レイヤーマスク | 非破壊編集 | 低 |

実装時の注意:

- Photoshop や Procreate の代替を目指さない。
- ゲーム素材を整えるための必要機能に絞る。
- 元画像を破棄しない。
- Undo / Redo を必ず維持する。

### 4.2 判定編集の細かさ

ゲーム素材としては、当たり判定が重要である。将来は次を検討する。

| 機能 | 目的 | 優先度 |
|---|---|---|
| 多角形判定 | 斜め形状、武器、地形 | 高 |
| フレーム別判定 | 攻撃・被弾タイミング | 高 |
| 左右反転時の判定 | アクションゲーム向け | 高 |
| sensor 判定の見える化 | 取得範囲、接地判定 | 中 |
| 判定プリセット | player / enemy / item / projectile | 中 |
| 判定エクスポート単位の選択 | Phaser / Pixi / Canvas 用補助 | 中 |

実装時の注意:

- 既存の rect / circle を壊さない。
- 多角形を追加する場合は Schema と docs を先に更新する。
- 物理エンジン依存の形にはしない。

### 4.3 アニメーション制作の細かさ

現在はフレームと簡易リグがある。将来は次を検討する。

| 機能 | 目的 | 優先度 |
|---|---|---|
| Onion skin | 前後フレーム確認 | 高 |
| フレーム別判定表示 | 攻撃・被弾確認 | 高 |
| タイムラインのズーム | 長いアニメーション編集 | 中 |
| キーフレーム複製 | 手作業削減 | 高 |
| イージング | 簡易リグの動き改善 | 中 |
| 左右反転アニメーション生成 | 歩行・攻撃の再利用 | 高 |
| 表情差分 | キャラクター表現 | 中 |

実装時の注意:

- Spine / Rive 互換を名乗らない。
- 簡易リグはフレームへ焼き込む補助である。
- ランタイムボーンアニメーションを初期前提にしない。

## 5. 背景・タイル・ギミックの強化要件

### 5.1 背景

将来の背景機能は、絵を描くことより、ゲーム背景として使いやすくすることを重視する。

必要候補:

- ループ継ぎ目確認
- 遠景 / 中景 / 近景のプレビュー
- parallaxSpeed の実再生
- カメラ移動プレビュー
- 背景レイヤーごとの書き出し
- 背景全体の合成書き出し

### 5.2 タイル

タイルは、単体素材だけでなく、タイルセットとして扱えるようにする必要がある。

必要候補:

- tile set editor
- grid slicing
- tile collision map
- one way platform 表示
- hazard 表示
- visualType の候補管理
- tile atlas padding
- tile preview

### 5.3 ギミック

ギミックは、画像とタグだけでは不十分である。将来は動きの確認が必要になる。

必要候補:

- movementPreset のプレビュー
- horizontal / vertical / rotate / pendulum のパラメータ
- 当たり判定との連動
- sensor 範囲
- start delay
- loop / ping-pong

実装時の注意:

- ゲームエンジンの実行環境を作り込まない。
- まずは見た目とメタデータのプレビューに留める。

---

## 6. エフェクト最小拡張要件

effect アセットは、Phase 17 で最低限対応した後、将来は次を検討する。

| 機能 | 目的 | 優先度 |
|---|---|---|
| 発光風プレビュー | hit / aura 表現 | 中 |
| trail 表示 | 斬撃、弾跡 | 中 |
| フレームごとの blendMode | 発光アニメ | 中 |
| エフェクトの発生アンカー | weapon / hit / center | 高 |
| duration と animation の整合チェック | 再生時間管理 | 高 |

禁止:

- 初期段階で GLSL / WebGPU シェーダー必須にしない。
- パーティクルエンジンを急に入れない。
- 既存のフレームアニメーションを壊さない。

---

## 7. 書き出し品質の要件

将来、書き出しの品質を上げる。

必要候補:

- Atlas padding
- extrude
- trim の有無
- フレーム名規則チェック
- 1x / 2x / 3x 出力
- 透明ピクセルの扱い
- WebP 品質設定
- PNG 最適化
- helper snippet の選択出力
- engine guide の選択出力

ただし、export preset 管理 UI は今回の対象外である。必要になるまでは、まず固定設定または最小のオプションに留める。

---

## 8. 3D Asset Preparation Mode の入口要件

3D 対応は、生成 AI から始めない。

最初の要件は次である。

- GLB / glTF を読み込める。
- 3D モデルを表示できる。
- ファイルサイズを表示できる。
- ポリゴン数を表示できる。
- テクスチャ枚数と解像度を表示できる。
- アニメーション有無を表示できる。
- 中心点、足元基準、向きを確認できる。
- 3D 用メタデータを保存できる。
- 3D 生成 AI との連携は後回しにする。

詳細は `THREE_D_ASSET_PREPARATION_REQUIREMENTS.md` に定義する。

---

## 9. docs とテスト要件

将来拡張では、次を必ず守る。

- 型を追加したら `docs/DATA_FORMAT.md` を更新する。
- ZIP 構成を変えたら `docs/EXPORT_FORMATS.md` を更新する。
- engine 連携を変えたら `docs/ENGINE_INTEGRATION.md` を更新する。
- E2E を追加する。
- 既存 E2E を弱くしない。
- README の Phase 表示を実態と一致させる。

---

## 10. 完了判定

この文書に基づく将来拡張は、次を満たした場合に完了扱いにできる。

- 既存 Phase 1〜17 の機能を壊していない。
- 新しい機能に Unit test がある。
- UI を伴う機能には E2E がある。
- docs が更新されている。
- `npm run build` が成功する。
- `npm run test` が成功する。
- 関係する場合は `npm run e2e` が成功する。
- `npm run lint` と `npm run format:check` が成功する。

---

## 11. 注意点

この文書は「全部をすぐ実装するリスト」ではない。

優先順位は、既存の制作体験を壊さないこと、書き出し品質を上げること、ユーザーがゲームに組み込みやすくすることの順で決める。

販売や SaaS 化の議論は、ここでは扱わない。

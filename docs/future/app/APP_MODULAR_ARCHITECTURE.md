## 1. 中心命題ふたたび：多機能と軽量の両立

`APP_PRODUCT_VISION.md` 3 章の矛盾——「機能は多くないと使われない／同梱物は少なくないと重くて使われない」——を解く設計を、ここで具体化する。原理は 1 つ。

> **プロダクトは機能を「持つ」が、利用者は機能を「入れる」。重さはインストールした人だけが負う。**

これを成立させるには、次の 4 つが技術的に保証されねばならない。

1. **未インストール機能はゼロコスト**（起動時間・常駐メモリ・バイナリサイズに一切乗らない）。
2. **インストールは可逆**（入れる／外すが安全で、外せば重さも戻る）。
3. **コアは機能ではなく契約**（データ・保存・書き出し・拡張点の土台だけを持ち、痩せている）。
4. **モジュール境界がデータ互換性を壊さない**（どの構成でも同じ `project` を開ける）。

## 2. レイヤー構成（Core / Runtime / Modules / Catalog）

```txt
┌─────────────────────────────────────────────────────────────┐
│ Shell（ネイティブ外殻：ウィンドウ・OS 連携・ファイル・GPU）        │  ← 痩せた常駐部
├─────────────────────────────────────────────────────────────┤
│ Core Contract（不変の土台）                                     │
│  ・Project / Document モデルと保存形式（バージョン付き）           │
│  ・非破壊レイヤ（source 保持・派生分離）                          │
│  ・座標／単位／anchor／collider の意味論（2D と 3D で別型）        │
│  ・Undo/Redo と履歴経路                                         │
│  ・Validation / Inspection フレーム                             │
│  ・Export フレーム（README / import notes 同梱の器）              │
│  ・Extension Host（モジュールの登録・権限・ライフサイクル）        │
├─────────────────────────────────────────────────────────────┤
│ Edition Runtime（2D Runtime ／ 3D Runtime：どちらか一方を起動）   │
│  ・2D: raster/vector canvas・frame・tile・atlas の基盤          │
│  ・3D: scene graph・mesh/material・inspection・軽量化の基盤       │
├─────────────────────────────────────────────────────────────┤
│ Feature Modules（利用者が選んで入れる機能パック）                  │  ← 入れた分だけ重い
│  例) Tileset Pro / Skeletal Animation / Atlas Packer /          │
│      Collider Studio / 3D Optimize / VRM Kit / Retarget /       │
│      Palette & Dithering / Batch & Variants / Format Bridges …  │
├─────────────────────────────────────────────────────────────┤
│ Catalog（機能カタログ：ローカル定義。オンライン取得は任意）         │  ← 一覧は軽い。実体は未取得
└─────────────────────────────────────────────────────────────┘
```

- **Shell** と **Core Contract** はどの利用者も共通で、意図的に痩せさせる。ここに機能を足さない。
- **Edition Runtime** は 2D か 3D の一方だけをプロセスに載せる。2D 利用者は 3D の重い描画依存を 1 バイトも起動しない（逆も同様）。
- **Feature Modules** はインストール時に初めて実体が入る。未インストールは Catalog 上の「入れられる候補」でしかない。
- **Catalog** は軽量な定義（名前・説明・要件・依存・サイズ・ライセンス）だけを持ち、機能の重い実体とは分離する。

## 3. モジュールの形：マニフェスト契約

各モジュールは、実体（コード・アセット）と分離した **manifest（契約）** を持つ。manifest は Catalog に載る軽量メタで、次を宣言する（形式は概念例）。

```jsonc
{
  "id": "com.chameleon.module.skeletal-animation",
  "title": "Skeletal Animation",
  "edition": "2d",                       // 2d | 3d | shared
  "version": "1.0.0",
  "coreContract": ">=1.0 <2.0",          // 互換な Core 契約範囲
  "dependsOn": ["com.chameleon.module.timeline"],  // モジュール依存
  "provides": ["panel:rig", "tool:bone", "export:spine-notes"],
  "capabilities": ["read-project", "write-derived", "gpu-optional"],
  "footprint": { "installMB": 14, "residentMB": "lazy", "startupMs": 0 },
  "license": { "spdx": "TBD", "commercialUse": "TBD", "verified": false }
}
```

契約が保証すること:

- **`coreContract` 範囲外は入れられない**。Core が壊れる組み合わせを事前に弾く。
- **`dependsOn` は自動解決**。単体で意味を成さないモジュールは依存を連れてくる（利用者は最終的な合計サイズ・追加権限を見て承認する）。
- **`footprint.startupMs: 0` を原則**。モジュールは起動を遅くしない。実体ロードは初回利用まで遅延（lazy）。
- **`license.verified: false` は導入不可**。ライセンス・商用利用条件が未確認のモジュールは Catalog に「未検証」表示され、インストール導線を出さない（現行 CLAUDE.md のライセンス規律を継承）。
- **`capabilities` は最小権限**。project 読み取り・派生書き込み・GPU 使用などを明示し、Extension Host が境界を強制する。

## 4. install-on-demand の体験と可逆性

利用者の体験を「使われない理由」に照らして設計する。

1. **最小で速く始まる**: 初回起動はコア + 選んだ Edition Runtime のみ。空のプロジェクトを迷わず作れる。
2. **足りない時にだけ勧める**: 例えば「ボーンで動かしたい」操作に触れた時、必要な `Skeletal Animation` モジュールを **その場で** 提示する（未インストール機能を UI に常時ちらつかせない）。
3. **合計コストを見せて入れる**: インストール前に、追加サイズ・追加常駐メモリ見込み・必要権限・依存・ライセンス状態を提示し、利用者が承認する。
4. **外せば戻る**: アンインストールで実体・常駐・起動コストが戻る。**プロジェクトデータは壊れない**（5 章）。
5. **プロファイルで束ねる**: 「ドット絵制作」「tileset 制作」「3D 検品」等の推奨プリセットを用意し、個別選択が面倒な利用者は 1 クリックで妥当な最小セットを入れられる。

この体験は、既存の成功例（VS Code の拡張マーケット、Blender の extensions、Godot AssetLib、Bevy の plugin 合成）から学ぶ。特筆すべき教訓として、Krita 本体（GitHub 10,072★）に対し単一のサードパーティ拡張 `krita-ai-diffusion` が 10,354★ を集めた事実は、**適切な拡張機構が本体以上の価値・熱量を生みうる**ことを示す（出典は `APP_COMPETITIVE_LANDSCAPE.md`）。ただし本コンセプトの拡張は、まず**公式モジュール**から始め、第三者拡張の開放は後段の判断とする（8 章 open items）。

## 5. データ互換性の不変条件（構成に依存しない）

モジュール構成は自由でも、**プロジェクトは壊れてはならない**。次を不変条件（invariant）とする。

- **どのモジュール構成でも、同じ `project` を開ける**。あるモジュールで付けたデータ（例: rig, collider, 3D anchor）は、そのモジュール未インストールの環境でも **保持され、破壊されず、無視されるだけ**（forward/unknown-data 尊重。現行 ADR-0012 の思想を継承）。
- **モジュールは Core のデータ契約を拡張できるが、既存フィールドの意味を変えられない**。名前空間つきの拡張領域にのみ書ける。
- **保存形式はバージョン付き**。migration は Core が持ち、モジュールは migration を勝手に書き換えない。
- **UI state / 一時状態は保存データに混ぜない**（現行 CLAUDE.md 6 章の規律を継承）。
- **export は Core の器を通す**。モジュールは preset と import notes を足せるが、ZIP 構造・README 同梱の原則を壊さない。

これにより「必要な時だけモジュールを入れて作業し、外して配布し、別環境で開き直す」が安全に回る。

## 6. ネイティブ外殻の方向性（未採用の検討材料）

「軽量が前提」という要求は、外殻技術の選定を左右する。ブラウザ制約を外す一方で、**重い外殻を選んで振り出しに戻ってはならない**。

2026-07 時点の外部実測（`APP_COMPETITIVE_LANDSCAPE.md` に出典）を材料として置く。

| 観点 | Chromium 同梱型（例: Electron） | システム WebView + ネイティブコア型（例: Tauri） |
|---|---|---|
| インストーラサイズ | 概ね 80–150MB | 10MB 未満の報告 |
| 常駐メモリ（アイドル） | 概ね 150–300MB | 概ね 30–50MB |
| 冷間起動 | 1–2 秒程度 | 0.5 秒未満の報告 |
| 実移行事例 | — | Hoppscotch が 165MB→8MB・メモリ約 70% 減 |

含意（**方向であって決定ではない**）:

- 「軽量が前提」を満たすには、**外殻は痩せた方式を優先検討**する価値が高い。ただし採用は、対象機能（canvas 描画・3D 描画・大画像処理）の実測、開発言語（Rust 等）の体制、OS 別 WebView 差の検証を経てからにする。
- 描画は Edition ごとに分ける。2D は 2D 向け描画（現行資産の PixiJS 等の知見が活きる）、3D は 3D 描画（three.js / Babylon.js / Filament 等の候補）を、**Edition Runtime のロード時にだけ**持ち込む。どちらの重い依存も、他方の利用者に負わせない。
- 重い処理（大画像・atlas packing・mesh 簡略化・テクスチャ圧縮）は、**Worker / 別スレッド / ネイティブ側**へ逃がし、UI を止めない。ネイティブ化の最大の見返りはここにある（真の並列・SIMD・GPU compute・メモリ制約からの解放）。

具体的な候補ライブラリ（meshoptimizer / draco / compressonator / assimp / three-vrm 等）の評価は Catalog モジュール単位で行い、**採用前にライセンス・商用利用・bundle・実測を必ず確認**する。

## 7. 性能規律（軽量性を「機能」でなく「前提」にする）

軽量性は後付けの最適化タスクではなく、**受け入れ基準**として最初に固定する。以下は概念上の budget 例（実機・実測で確定する対象。数値は仮）。

| 指標 | 最小構成の狙い | 破ってはいけない原則 |
|---|---|---|
| 冷間起動 | 1 秒未満級 | モジュールを増やしても最小構成の起動は劣化させない |
| 常駐メモリ（空プロジェクト） | 数十 MB 級 | 未インストール機能は 1 バイトも常駐しない |
| 入力応答 | 1 フレーム内の手応え | 重い処理は必ず非同期・別スレッドへ逃がす |
| インストーラ（最小） | 小さく保つ | Edition 依存・モジュール実体は同梱しない |

規律ルール:

- **未使用ゼロコストの原則**: 使わない機能が起動・常駐・サイズに乗っていたら、それは設計上の欠陥として扱う。
- **計測が先、機能が後**: 各モジュールは footprint を宣言し、リリース判定で実測と突き合わせる。
- **退行を検知する**: 起動時間・常駐メモリ・最小バイナリサイズは継続計測し、悪化を CI/レビューで止める。
- **重い依存は Edition/モジュール境界の内側に閉じる**: Core と Shell へは絶対に染み出させない。

数値目標・計測環境・退行検知の具体は `APP_ROADMAP_DECISIONS_OPEN_ITEMS.md` の Gate に落とす。

## 8. 未決事項（この文書が決めないこと）

- 外殻技術・描画ライブラリ・言語の**採用**（実測・ライセンス・体制の確認後）。
- モジュール実体の配布方式（同梱プリセット vs オンライン取得 vs ローカルパッケージ）。オンライン取得を選ぶ場合の署名・検証・オフライン動作の要件。
- 第三者拡張の開放可否と、その場合の権限モデル・審査・サンドボックス強度。
- 保存形式が現行 `.casproj` / `asset.json` を継承するか、新形式にするか（互換方針は別途決定）。
- 価値の位置づけ（`APP_PRODUCT_VISION.md` 5 章）を、実際の提供形態にどう写すか（本コンセプトは販売を目的にしない）。

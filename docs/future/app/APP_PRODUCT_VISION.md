## 1. なぜ「別物」として構想するか

現行の Chameleon Asset Studio は、ブラウザ内で完結するローカル制作ツールとして、意図的に制約を受け入れて設計されてきた。IndexedDB の容量枠、単一 origin のメモリ上限、WebGL 前提、File System Access の限界、真のマルチスレッド／SIMD／GPU compute への制御の弱さ、ネイティブなファイルダイアログや OS 連携の不在——これらは「安全に、軽く、どこでも動く」ための正しい割り切りだった。

一方で、本コンセプトが目指すのは別の頂点である。**ブラウザ制約を前提にしない、目的特化のネイティブ制作アプリ**として、2D と 3D をそれぞれ最適な形で提供する。ここでは「ブラウザだからできない」を理由にした妥協を一度すべて外し、代わりに「ネイティブだからこそ守るべき規律（軽量性・最適化・モジュラー性）」を最上位の設計原理に置く。

この構想を現行製品と混ぜない理由は、現行の 2D Pro Gate / 3D ロードマップの完成を妨げないためである。知見は引き継ぐが、制約と目標は引き継がない。

## 2. プロダクトの一文定義

> **Chameleon Asset Studio App** は、2D と 3D のゲーム用アセットを「作る・整える・検品する・書き出す」ための、**軽量なコアに、利用者が必要な機能だけを後から足していけるネイティブ制作アプリ**である。

- **ネイティブ**: ブラウザのサンドボックス制約を前提にしない。OS のファイル・メモリ・並列処理・GPU を正面から使う。
- **目的分離**: 2D 制作と 3D 整備は、同じ思想・同じ操作感を共有しつつ、別のエディション（プロファイル）として最適化する。1 つの肥大アプリに混ぜない。
- **モジュラー**: 最小構成で軽く起動し、利用者が自分のニーズに合わせて機能パックを選んでインストールする。使わない機能の重さを、使う人に負担させない。

## 3. 「使われない理由」から設計する（非採用の法則）

本コンセプトの出発点は、機能一覧ではなく **「制作ツールが使われなくなる理由」** である。利用者は次のいずれか 1 つでも当てはまると離れる。

| # | 使われない理由 | 本コンセプトの応答 |
|---|---|---|
| 1 | 重すぎる（起動が遅い・メモリを食う・もたつく） | 軽量性を機能ではなく**前提**にする。コアは最小。未インストール機能はゼロコスト |
| 2 | 他ツールと機能が同じ（差が無い） | 「作る→整える→検品→書き出す」の一気通貫と、検品・互換性の確かさで差を出す |
| 3 | 機能が少ない（やりたいことが足りない） | カタログとしては広い。ただし**全部を同梱しない**。必要な人が必要な機能を足せる |
| 4 | 使い勝手が悪い（操作が迷う・手順が多い） | 1 アセットを迷わず完了できる導線。Undo/Redo と再開性を最初から通す |
| 5 | UI が悪い（見て分からない・崩れる） | 2D/3D で一貫した見た目と用語。エディション間で操作を転用できる |
| 6 | UX が悪い（体験として不快・不安） | 失敗の理由が文章で分かる。元データを壊さない。手動で必ず回復できる |

ここに **根本の矛盾** がある。

- 「機能が少ないと使われない」→ 機能は**多く**なければならない。
- 「重すぎると使われない」「使わない機能で重いと離れる」→ 同梱物は**少なく**なければならない。

多機能と軽量は、単一の同梱アプリでは両立しない。**この矛盾を解く唯一の設計が「モジュラー install-on-demand」である。** プロダクトは機能を多く**持つ**（カタログ）が、利用者は使う機能だけを**入れる**（インストール）。重さは選んだ人だけが負う。これが本コンセプトの中心命題であり、詳細は `APP_MODULAR_ARCHITECTURE.md` が担う。

## 4. 対象利用者とエディション

### 4.1 2D Edition（作る・整える中心）

対象: 個人〜小規模のゲーム制作者、ドット絵／イラスト系アセット制作者、tileset / sprite sheet を扱う人。

中核体験: 空キャンバス・テンプレート・画像取り込み・図形・パーツから作り、レイヤー／フレームを整理し、原点・アンカー・当たり判定を付け、フレームアニメーションを扱い、検品して、外部エンジンへ持ち込める形に書き出す。

### 4.2 3D Edition（整える・検品中心）

対象: 外部（配布素材・生成物・DCC ツール出力）で作られた 3D モデルを、ゲームで使える状態へ整えたい制作者。

中核体験: GLB / glTF を安全に読み込み、壊れ・重さ・スケール・向きを検品し、原点／足元／向き／bounds を整え、3D アンカーと 3D 当たり判定を付け、元を壊さずに軽量化候補を作り、metadata・inspection report つきで書き出す。3D 本体を「作る」道具（Blender 級モデリング）にはしない。

### 4.3 共有する土台（同じ思想・別最適化）

Home / Project Dashboard、保存・再開の考え方、元データ非破壊の原則、metadata を UI・export から独立させる設計、schema 検証、README / import notes を同梱する書き出し方針、検品（validation / inspection）の思想、そしてモジュール機構そのもの。**編集画面と座標系・データモデルは分ける。**

## 5. 価値の位置づけ（value positioning）

「月額 2,000 ドル / 買い切り 10,000 ドル級」は価格ではなく、**どの品質帯を狙うかの目安**である。この帯は、既存プロツールの外部公開・調達データ（2026-07 時点、`APP_COMPETITIVE_LANDSCAPE.md` に出典）から見ると、概ね次に相当する。

- サブスク帯: SideFX Houdini FX のサブスクリプション（年額 ≈ $4,495）と Core（年額 ≈ $1,995）の中〜上。
- 買い切り帯: Esoteric Spine の上位ライセンス（調達データで 1 件 $7,000〜$11,000 規模）や Houdini 系の perpetual + maintenance に相当。

つまり狙いは「安価な便利ツール」ではなく、**プロの制作パイプラインで対価を説明できる確かさ**である。ただし本コンセプトでは、その価値を **課金・アカウント・クラウドではなく、制作物の確かさ・軽さ・拡張性で説明する**方針を採る（販売は目的にしない）。

補足として、adoption（愛される度）の観点では対極の事例も学ぶ。Aseprite は $19.99 の低価格ながら GitHub 38,000★ 超を集める。焦点を絞った・速い・スクリプト拡張できるネイティブツールは、価格に関係なく強い支持を得る。**高価値帯を狙いつつ、Aseprite 的な「軽くて焦点が明快」を捨てない**——この両立が本コンセプトの品質目標である。

## 6. 現行 Chameleon から引き継ぐ知見（系譜）

本コンセプトは白紙ではない。現行リポジトリで積み上げた次の知見を土台にする（**制約ではなく知見として**）。

- 非破壊編集と、UI state / 保存データ / export の責務分離（`docs/adr/0007-data-layer-separation.md`）。
- 座標・アンカー・当たり判定・アニメーションの意味論と、その ADR 化（`docs/adr/`）。
- 保存形式・migration・forward compatibility の契約設計（`2D_ASSET_DATA_CONTRACT.md`, `3d/3D_ASSET_DATA_CONTRACT.md`）。
- export ZIP に README / import notes を同梱し、まず直接連携ではなくファイルで渡す方針。
- 3D は「作る」ではなく「整える」から始める判断（`PRODUCT_DIRECTION_2D_TO_3D.md`, `3d/README.md`）。
- docs-first・段階 Gate・証拠つきレビューの運用文化。

引き継がないもの: ブラウザ完結の前提、IndexedDB 容量枠、単一 origin メモリ制約、WebGL 限定、no-dependency の硬い縛り、単一アプリに全機能を同梱する形。これらは本コンセプトでは**設計上の与件ではない**。

## 7. 非目標（この構想でやらないこと）

- 3D 生成 AI をアプリ本体に内蔵すること。3D 生成は外部処理の候補として扱う。
- Blender / Maya 級の本格モデリング・本格リグ制作・本格アニメ制作の代替。
- 総合ゲームエンジン化（実行時ランタイム・ゲームロジック・物理シミュレーションの本体化）。
- クラウド前提・アカウント必須・常時オンライン必須の設計。ネイティブでローカル完結を既定にする。
- ライセンス未確認の依存や研究モデル重みの同梱。
- 「モジュール化のためのモジュール化」。分割は軽さ・保守性・利用者選択のためであり、目的化しない。

## 8. 成功条件（コンセプトとしての合格ライン）

このコンセプトが「作る価値がある」と言えるのは、次がすべて説明できるときである。

1. 最小構成が、同種の重量級ツールより明確に速く・軽く起動する（性能規律は `APP_MODULAR_ARCHITECTURE.md` 7 章参照）。
2. 利用者が、自分のニーズに対して過不足なく機能を選んで入れられ、使わない機能の重さを負わない。
3. 2D と 3D が、別最適化されながら同じ操作感で行き来できる。
4. 1 つのアセットを、作成／整備から検品・書き出しまで迷わず完了できる。
5. 書き出した成果物が、外部エンジンで「どう使うか」まで含めて確かである。
6. 上記を、課金・クラウドに頼らず、制作物と体験の質だけで説明できる。

各条件の検証観点は `APP_ROADMAP_DECISIONS_OPEN_ITEMS.md` に落とす。

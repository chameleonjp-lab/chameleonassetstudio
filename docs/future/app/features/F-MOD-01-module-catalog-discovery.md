## 0. 完成系での位置づけ（逆算リンク）

由来する完成系（`features/README.md` 2 章）:

- **S-DONE-2**: **最小構成で軽く起動**し、**必要な機能だけをインストール**して広げられる。使わない機能の重さを負わない。

逆算の理由: 「必要な機能だけを選んで入れる」には、まず **入れられる機能を、重さ・依存・権限・ライセンスまで含めて一覧・発見できる入口**が要る。しかもその一覧自体が軽くなければ（実体を読み込んでいたら）矛盾する。本機能は、モジュール実体を読み込まずに軽量メタだけで一覧・検索・文脈提示する **Catalog と発見**を定義する。install / uninstall そのものは F-MOD-02 が担う。

関係マップのノード: `A5 Catalog`（本機能が実現）、`A6 Manifest 契約`（メタの出所）、`P3 install-on-demand`（実現）、`P4 未使用ゼロコスト`（閲覧もゼロコスト）、`G1 ライセンス検証 gate`（検証状態の表示）。

## 1. ユーザーストーリー

- 制作者として、**どんな機能を後から足せるか**を、重さ・必要権限・ライセンスまで見て選びたい。
- 制作者として、**今の作業に関係ない機能で画面が散らからない**でほしい（2D 作業中に 3D 専用機能が邪魔しない）。
- 制作者として、**「その操作にはこの機能が要る」**とその場で気づけると迷わない。
- 制作者として、**一覧を眺めるだけで重くならない**と安心して探せる。

## 2. 用語

- **Catalog**: 入れられるモジュールの一覧。**軽量メタ（manifest 由来）だけ**を持ち、モジュール実体とは分離する。
- **Manifest メタ**: title / 説明 / edition / version / dependsOn / footprint / capabilities / license・verified（`A6`、Arch 3 章）。
- **発見（Discovery）**: 閲覧・検索・絞り込み・文脈提示でモジュールに気づく手段。
- **実体（実装・アセット）**: 実際に重さを持つ本体。**インストールするまで取得・ロードしない**（F-MOD-02）。

## 3. 機能要件（FR）

### 3.1 軽量カタログ

- **FR-1** Catalog は各モジュールを**軽量メタだけ**で一覧する（title / 説明 / edition / version / dependsOn / footprint / capabilities / license・verified）。実体は含めない。
- **FR-2** Catalog の閲覧は**モジュール実体を取得・ロードしない**。一覧を眺めても起動・常駐・サイズが増えない（`P4` ゼロコスト）。
- **FR-3** 各エントリは次を提示する: 何をするか、**footprint（installMB / residentMB / startupMs）**、依存、必要権限（capabilities）、**ライセンスと verified 状態**、インストール済み/未の別。

### 3.2 発見

- **FR-4** 閲覧・**検索**・**絞り込み**（edition / カテゴリ / 済み・未）ができる。現在のエディション文脈に無関係なモジュールで一覧を散らかさない。
- **FR-5** **文脈連動の発見**: 未インストール機能が要る操作に触れたとき、その場で該当モジュールを提示する（`APP_MODULAR_ARCHITECTURE.md` 4 章）。
- **FR-6** メイン UI に未インストール機能の宣伝を**常時ちらつかせない**（発見は文脈時・カタログ内に限る）。

### 3.3 信頼・状態

- **FR-7** **verified=false のモジュールは「未検証」表示**し、**インストール導線を出さない**（enforcement は F-MOD-04。本機能は表示と導線抑止）。
- **FR-8** **インストール済み / 未**を明確に区別し、現在の導入状態を反映する（F-MOD-02 と連動）。
- **FR-9** Catalog の出所は**既定でローカル**（同梱定義）。オンライン取得を採る場合は、署名・検証・オフライン動作の要件を宣言してからにする（未決。F-MOD-02 と連動）。

## 4. 非機能要件（NFR）

- **NFR-1（ゼロコスト閲覧）**: カタログ閲覧が実体をロードせず、起動・常駐を悪化させない（`P4`、FR-2）。
- **NFR-2（性能）**: 多数のモジュールがあっても一覧・検索が快適。
- **NFR-3（オフライン）**: ローカルカタログは完全オフラインで機能する。
- **NFR-4（判断可能性）**: インストール前に、**重さ・依存・権限・ライセンス**を見て損得を判断できる（FR-3）。
- **NFR-5（信頼）**: verified 状態が明確で、未検証は誤って入れられない（FR-7）。

## 5. データ・保存への影響

- Catalog メタ・**導入状態はアプリ環境レベル**の情報で、**プロジェクト保存の器（F-CORE-05）に含めない**。
- プロジェクトが「どのモジュールの名前空間データを含むか」は**プロジェクト側**の情報（F-CORE-05 の `D4` 名前空間・未知データ）。この 2 つ（アプリの導入状態 / プロジェクトのデータ）を混同しない。
- 未インストールのモジュールが付けたデータを含むプロジェクトを開いても、Catalog は「その機能が未導入」と示せるが、**データは F-CORE-05 の往復で保持**される（S-DONE-5 と整合）。

## 6. 依存機能

- **A6 / Manifest 契約**（Arch 3 章、将来 `APP_MODULE_MANIFEST_SPEC.md`）: 一覧するメタの出所。
- **F-MOD-02**（導入/アンインストール）: 実体取得とインストール状態（FR-8）。Catalog はその入口。
- **F-MOD-03**（依存解決/footprint 開示）: 依存と合計コスト提示の詳細（Catalog は単体メタ、合計は導入時）。
- **F-MOD-04**（ライセンス検証ゲート）: verified の enforcement（FR-7）。
- **F-CORE-09 / A9**（Extension Host）: 導入済みモジュールの把握。

## 7. 受け入れ条件（Acceptance）

- **AC-1**: Catalog を閲覧しても、モジュール実体がロードされない（起動・常駐メモリが閲覧で増えない）（NFR-1）。
- **AC-2**: 各エントリが footprint・依存・権限・ライセンス/verified・導入状態を示す（FR-3）。
- **AC-3**: edition で絞り込むと、無関係エディションのモジュールが一覧から外れる（FR-4）。
- **AC-4**: verified=false のモジュールが「未検証」と示され、インストール導線が出ない（FR-7）。
- **AC-5**: 未インストール機能が要る操作に触れると、その場で該当モジュールが提示される（FR-5）。
- **AC-6**: メイン UI が未インストール機能の宣伝で常時散らからない（FR-6）。
- **AC-7**: ローカルカタログが完全オフラインで機能する（NFR-3）。

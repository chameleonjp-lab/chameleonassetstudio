## 0. 完成系での位置づけ（逆算リンク）

由来する完成系（`features/README.md` 2 章）:

- **S-DONE-2**: 最小構成で軽く起動し、**必要な機能だけをインストール**して広げられる。**使わない機能の重さを負わない。**

逆算の理由: カタログ（F-MOD-01）で見つけた機能を、**合計コストを見て入れ**、**外せば重さが戻り**、しかも**外してもプロジェクトを壊さない**——この「入れる／外すの実体」が、install-on-demand の中核。本機能は導入・撤去の可逆性と**コスト戻り**を定義する。

関係マップのノード: `P3 install-on-demand`（実現：導入半分）、`P4 未使用ゼロコスト`（撤去でコスト戻り）、`F-MOD-01`（入口）、`D3 構成非依存`（撤去してもデータ非破壊）、`A9 Extension Host`（有効化/無効化）、`G1 ライセンス gate`（未検証は入れられない）。

## 1. ユーザーストーリー

- 制作者として、入れる前に**追加サイズ・常駐見込み・必要権限・依存・ライセンス**を見て納得してから入れたい。
- 制作者として、**使わなくなったら外して、重さ（サイズ・常駐・起動）が本当に戻る**と信じたい。
- 制作者として、機能を外しても、**その機能で付けたデータが載ったプロジェクトが壊れない**でほしい。
- 制作者として、入れ直せば**元どおり編集できる**と分かっていたい。

## 2. 用語

- **導入（Install）**: モジュール実体を取得し、機能を使える状態にすること。
- **撤去（Uninstall）**: 実体を外し、その重さ（サイズ・常駐・起動）を戻すこと。
- **合計コスト**: 導入で増える追加サイズ・常駐見込み・必要権限・連れてくる依存・ライセンス状態のまとまり。
- **遅延ロード（lazy）**: 導入は実体をディスクに置くだけで、実体のメモリ展開は**初回利用時**（起動時ではない）。

## 3. 機能要件（FR）

### 3.1 導入

- **FR-1** 導入前に、**合計コスト**（追加サイズ / 常駐見込み / 必要権限 capabilities / 連れてくる依存 / ライセンス・verified）を提示し、**利用者の明示承認**を得てから実行する。
- **FR-2** 導入は**起動コストを増やさない**。実体は初回利用時に**遅延ロード**し、起動時に載せない（`P4` / footprint.startupMs=0）。
- **FR-3** 導入は**ローカル同梱実体**、またはオンライン取得のいずれか。オンラインの場合は、**有効化前に完全性（署名・検証）を確認**する（方式は未決 `APP-O-02`）。

### 3.2 撤去とコスト戻り

- **FR-4** 撤去は実体を外し、**サイズ・常駐・起動コストを戻す**（残渣を残さない。`P4` / APP-RISK-02）。導入・撤去は**可逆**。
- **FR-5** 撤去は**プロジェクトを壊さない**。撤去したモジュールが付けたデータは、プロジェクト内で**保持され（往復）**、機能未導入の間は**非活性・非編集**になるだけ（`D3`、F-CORE-05 の未知データ保持）。
- **FR-6** **入れ直せば機能と編集可能性が復元**する。

### 3.3 安全性・依存・状態

- **FR-7** 導入・撤去は**安全で反復可能**。途中失敗（電源断・取得中断）でも、アプリとプロジェクトを**不整合状態にしない**（原子的に切り替え、失敗時は元へ）。
- **FR-8** **依存**: dependsOn を持つモジュールの導入は、必要な依存を連れてくる（解決の詳細は F-MOD-03）。合計コスト（FR-1）は**依存込み**で提示する。撤去は**逆依存を尊重**し、まだ必要とされる依存を巻き添えで消さない（詳細 F-MOD-03）。
- **FR-9** **verified=false は導入不可**（enforcement は F-MOD-04）。導入導線は verified のみ。
- **FR-10** **導入状態はアプリ環境レベルで永続**し、再起動後も保たれる。カタログ（F-MOD-01）へ反映する。プロファイル一括導入（F-MOD-05）も同じ導入機構に乗る。

## 4. 非機能要件（NFR）

- **NFR-1（コスト戻りの真正性）**: 撤去で起動・常駐・ディスクのコストが実際に戻る（残渣ゼロ。`P4`）。
- **NFR-2（可逆・安全）**: 途中失敗でもアプリ／プロジェクトを壊さない（FR-7）。
- **NFR-3（プロジェクト非破壊）**: 撤去がプロジェクトデータを失わせない（FR-5、NFR は絶対）。
- **NFR-4（性能）**: 導入・撤去が UI を固めない（非同期・別スレッド。F-PLT-03）。
- **NFR-5（オフライン/透明性）**: ローカル導入はオフラインで完結。オンライン取得は必要性を明示。合計コストと権限を実行前に開示（FR-1）。

## 5. データ・保存への影響

- **導入状態＝アプリ環境レベル**。プロジェクト保存の器（F-CORE-05）には含めない。
- **モジュールが付けたプロジェクトデータ**は、導入状態に関係なく保持（F-CORE-05 の `D4` 名前空間・未知データ往復）。撤去＝データ削除ではない。
- この分離（アプリの導入状態 / プロジェクトのデータ）を F-MOD-01 と同じく厳守する。

## 6. 依存機能

- **F-MOD-01**（カタログ）: 入口・合計コスト表示の素・導入状態の反映。
- **F-MOD-03**（依存解決/footprint）: 依存の連れ込み・逆依存の尊重・合計の算出（FR-8）。
- **F-MOD-04**（ライセンス検証）: verified enforcement（FR-9）。
- **F-CORE-05**（保存/未知データ）: 撤去してもデータ往復（FR-5）。
- **F-CORE-09 / A9**（Extension Host）: モジュールの有効化/無効化ライフサイクル。
- **F-PLT-02/03**（ネイティブ I/O・offload）: 実体取得・非同期化・原子的切替（FR-3/FR-7）。

## 7. 受け入れ条件（Acceptance）

- **AC-1**: 導入前に合計コスト・権限・依存・ライセンスが提示され、承認しないと導入されない（FR-1）。
- **AC-2**: 導入後もアプリ起動時間が変わらない（実体は初回利用でロード、起動では載らない）（FR-2）。
- **AC-3**: 撤去でディスク・常駐・起動コストが戻る（残渣が無いことを計測で示せる）（NFR-1）。
- **AC-4**: 撤去したモジュールのデータを含むプロジェクトが開け、保持・往復し、**入れ直すと編集可能性が復元**する（FR-5/FR-6）。
- **AC-5**: 導入・撤去を途中で中断しても、アプリ・プロジェクトが一貫状態に保たれる（FR-7）。
- **AC-6**: verified=false のモジュールに導入導線が無い（FR-9）。
- **AC-7**: 導入状態が再起動後も保たれ、カタログに反映される（FR-10）。

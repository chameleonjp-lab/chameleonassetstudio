## 0. 完成系での位置づけ（逆算リンク）

由来する完成系（`features/README.md` 2 章）:

- **D-DONE-8**: 保存し、開き直して**完全に再開**でき、クラッシュしても復旧できる。
- **S-DONE-5**: **どのモジュール構成でも同じプロジェクトを開け、未知データを壊さない。**
- **T-DONE-9**: 保存・再開でき、**元の 3D モデルを絶対に上書きしない**。

逆算の理由: 上記が成り立つには、保存の「器」が **版を持ち・古い版を安全に移行し・知らない版や知らないデータを壊さず往復できる** ことが要る。F-CORE-01 が「いつ・どう保存／再開するか」を担うのに対し、本機能は **器そのものの形式・版・移行の正本**（関係マップ `D1` 保存形式 / `D2` migration / `D3` 構成非依存 / `D4` 名前空間拡張）を定義する。

関係マップのノード: `D1` / `D2` / `D3` / `D4` / `A2 Core 契約`。

## 1. ユーザーストーリー

- 制作者として、**アプリを更新しても、前の版で保存したプロジェクトがそのまま開ける**と信じたい。
- 制作者として、**別 PC・別のモジュール構成で作ったプロジェクトを開いても、付いていたデータが消えない**でほしい。
- 制作者として、**移行（自動アップグレード）で何かが失われることは絶対に避けたい**。失敗しても元に戻せる安心が欲しい。
- 制作者として、**まだ入れていない機能が付けたデータ**を、うっかり削って保存したくない。

## 2. 用語

- **保存の器（Container）**: 1 プロジェクトを表す保存単位。Document・付与データ・参照素材・メタ・未知データを収める（物理形式は未決）。
- **format version**: 器の構造の版。互換判定に使う。
- **migration**: 古い version の器を、現在の version が解釈できる形へ移す処理。
- **予約領域 / 拡張名前空間**: Core が意味を固定する領域と、モジュールが自分の名前空間にだけ書ける領域（`D4`）。
- **未知データ**: 現在のアプリ／構成が解釈できないフィールド（新しい版・未インストールのモジュールの領域など）。

## 3. 機能要件（FR）

### 3.1 バージョニング

- **FR-1** 器は必ず **format version を保持**する。version 無しの器は「不明」として安全側（読み取り専用扱い）に倒す。
- **FR-2** version は **互換を判定できる規則**を持つ（例: 破壊的変更＝major、後方互換の追加＝minor。実際の採番方式は実装時に確定）。読み込み側は「読める／移行が要る／読めない」を version から判断できる。
- **FR-3** 予約領域（Core の意味が固定された部分）と拡張名前空間（`D4`）を分け、version はまず**予約領域**に対して定義する。

### 3.2 移行（migration）

- **FR-4** 開くとき、器の version が現在より古ければ、**現在の版へ in-memory で移行**してから使う（保存するまで元ファイルを書き換えない）。
- **FR-5** 移行は **段階的・順序付き**（v1→v2→v3…）で、各段は小さく検証可能にする（系譜: ADR-0015）。段を飛ばさない。
- **FR-6** 移行は **非破壊**。移行結果を保存して元ファイルを更新する前に、**元に戻せる退避（バックアップ）を必ず残す**（系譜: ADR-0006）。移行の途中失敗で、移行前ファイルを壊さない。
- **FR-7** 各移行は **決定的**（同じ入力→同じ出力）で、何を変えたかを記録できる（version 履歴・provenance。系譜: ADR-0013。記録の粒度は別途）。

### 3.3 未知・新しいデータの保持（往復）

- **FR-8** 器に **現在のアプリが解釈できないフィールド**（新しい minor 版の追加、未インストールのモジュールの名前空間データ）が含まれても、**保持し、保存時にそのまま書き戻す**（往復＝round-trip。`D3`/`D4`、系譜: ADR-0012）。
- **FR-9** 器の **major version が新しすぎて読めない**場合、**黙って下位版へ潰さない**。読み取り専用で開く／警告するなど、**破壊しない**経路にする（FR-6 と同じ非破壊原則）。
- **FR-10** モジュールは **自分の拡張名前空間にだけ**書ける。Core は名前空間境界を検証し、他領域や予約領域への越境を拒否する（`D4`）。

### 3.4 整合性・元素材

- **FR-11** 器は **検証可能**にする。読み込み時に構造検証を段階的に行い（系譜: ADR-0014）、壊れた器を安全に検出する。
- **FR-12** 器は **原子的保存と相性の良い形**にする（F-CORE-01 FR-8 / F-PLT-02）。部分書き込みを検出・拒否でき、直前の正常保存を上書き前に失わない。
- **FR-13** 取り込んだ **元素材（画像・3D モデル）を器が上書きしない**。元は分離して保持し、編集・軽量化は派生として持つ（`G5`、F-CORE-03）。

### 3.5 既存資産との関係

- **FR-14** 器が現行 `.casproj` / `asset.json` を **継承するか新設するか**は未決（`APP-O-01`）。本要件は**どちらでも成り立つ**ように書く。継承する場合は、現行 2D/3D のデータ契約（`docs/future/2D_ASSET_DATA_CONTRACT.md`, `docs/future/3d/3D_ASSET_DATA_CONTRACT.md`）との互換を別途定義する。

## 4. 非機能要件（NFR）

- **NFR-1（データ非損失の絶対性）**: 移行・保存・未知データ往復のいずれでも、データを失わない。移行の破壊的保存前に退避を残す（FR-6）。
- **NFR-2（往復忠実性）**: 未知・名前空間データは、意味を保ったまま往復する（欠落・改変しない）。
- **NFR-3（決定性）**: 同じ器＋同じアプリ版なら、移行結果は毎回同一（FR-7）。
- **NFR-4（構成非依存）**: どのモジュール構成でも器を開け、未対応データを保持する（`D3`）。
- **NFR-5（性能）**: 代表サイズの器の読み込み・移行が UI を固めない（非同期・別スレッド。F-PLT-03）。
- **NFR-6（オフライン）**: 版判定・移行・検証はローカル完結。
- **NFR-7（監査性）**: format version と適用された移行段が観測できる。

## 5. データ・保存への影響

- **本機能が `D1`（保存形式）・`D2`（migration）の正本**。器の論理構造・version 場所・予約/拡張名前空間・移行レジストリの契約はここに集約する。
- **未知データ保持と名前空間**（`D3`/`D4`）の規則をここで固定し、F-CORE-01（開く/保存）・F-CORE-09（Host）・各モジュールが従う。
- **物理形式は未決**（非目標）。要件は物理形式に依存しないよう書く。
- **UI 一時状態は器に入れない**（Arch 5 章の規律。F-CORE-01 6 章の分離と一致）。

## 6. 依存機能

- **F-CORE-01**（ライフサイクル）: 本器を使って開く/保存/再開する（相互。F-CORE-01 は本機能に依存）。
- **F-CORE-03**（非破壊）: 元素材分離・派生保持（FR-13）。
- **F-CORE-06**（検品/検証）: 段階的構造検証（FR-11）。
- **F-CORE-09**（Extension Host）: 名前空間の持ち主（モジュール）の把握。未インストールでも FR-8 の往復で成立。
- **F-CORE-02**（自動保存/復旧）・**F-PLT-02**（ネイティブ I/O）: 原子的保存・退避（FR-6/FR-12）。

## 7. 受け入れ条件（Acceptance）

- **AC-1**: v(N-1) の器を vN アプリで開くと正しく移行され、**移行前の復元可能な退避**が残る（FR-4/FR-6）。
- **AC-2**: 未インストールのモジュールの名前空間データを含む器を開き、保存し直すと、そのデータが**変わらず往復**する（FR-8/NFR-2）。
- **AC-3**: 新しい minor 版の器を開いても、追加フィールドを**落とさず**開ける（FR-8）。
- **AC-4**: 新しい major 版の器を、**黙って下位版へ潰さない**（読み取り専用/警告。FR-9）。
- **AC-5**: 壊れた/部分書き込みの器を検出し、**直前の正常保存を上書きしない**（FR-11/FR-12）。
- **AC-6**: いかなる保存・移行後も、取り込んだ**元素材ファイルがバイト一致**（FR-13/NFR-1）。
- **AC-7**: 同じ器を同じアプリ版で複数回移行して、結果が毎回一致（NFR-3）。

## 8. 非目標

- 物理保存形式（zip / sqlite / json+bin 等）の確定。実装判断であり、人間承認と実測が要る。
- クラウド保存・版履歴（history）・同期。
- 2D↔3D のエディション間変換。
- provenance 記録の詳細粒度（本機能は「記録できる」ことまで。詳細は別途）。

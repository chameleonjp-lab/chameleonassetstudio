## 1. 逆算の方法（完成系 → 機能 → 仕様・要件）

1. **完成系を言い切る**: 「利用者が最終的に何を、迷わず、確実にできる状態か」を、エディションごとに具体文で固定する（2 章）。
2. **各完成系文から機能を導く**: 「これが成り立つには何の機能が要るか」を問い、機能を洗い出す（3 章のインベントリ）。
3. **機能ごとに仕様・要件を書く**: 4 章のテンプレートで、機能要件（FR）・非機能要件（NFR）・データ影響・依存・受け入れ条件・非目標・関係マップ linkage を埋める。
4. **上流から詰める**: 依存の土台（Core / 保存 / モジュール機構）を先に、次に各エディションの制作パイプラインを段の順に詳細化する（5 章）。
5. **地図を更新する**: 機能を詳細化するたびに関係マップのノード粒度・エッジ・カバレッジを更新し、抜けを可視化し続ける。

## 2. 完成系の定義（逆算の起点）

上位は `APP_PRODUCT_VISION.md` 4・8 章。ここでは逆算に使える粒度で「done」を言い切る。すべて**オフライン・軽量ネイティブ・非破壊・必要な機能だけ導入**が前提。

### 2.1 2D Edition の完成系

利用者は、軽量なネイティブアプリで、次を迷わず・確実にできる。

- **D-DONE-1**: 空キャンバス／テンプレート／画像取り込み／図形／パーツから 2D アセットを新規作成できる。
- **D-DONE-2**: レイヤー・パーツ・フレームを整理し、ラスタ／ベクタを非破壊で編集・修正できる。
- **D-DONE-3**: 原点・アンカー・当たり判定（rect / circle / polygon）を付けられる。
- **D-DONE-4**: フレームアニメーションと簡易リグ焼き込みを扱える。
- **D-DONE-5**: tileset / sprite sheet を再整理し、atlas に詰められる。
- **D-DONE-6**: 「エンジンで使える状態か」を検品できる（命名・サイズ・原点・atlas 整合）。
- **D-DONE-7**: PNG / WebP / Sprite Sheet / Atlas / JSON / ZIP を、対象別 preset と import notes つきで書き出せる。
- **D-DONE-8**: プロジェクトを保存し、開き直して**完全に再開**でき、クラッシュしても復旧できる。

### 2.2 3D Edition の完成系

- **T-DONE-1**: GLB / glTF（+bin+textures）を**安全に**読み込める（壊れ・危険を検知）。
- **T-DONE-2**: よくある他形式（OBJ 等）も取り込める。
- **T-DONE-3**: ファイルサイズ・ポリゴン数・マテリアル・テクスチャ・bounds・妥当性・VRM を検品できる。
- **T-DONE-4**: 原点・足元・向き・スケールを整え、3D アンカーと 3D 当たり判定（box / sphere / capsule）を付けられる。
- **T-DONE-5**: 骨格を検査し、「右手・頭」等の対応付けができる。
- **T-DONE-6**: **元を壊さず**軽量化した別モデルを作り、前後を比較して選べる。
- **T-DONE-7**: metadata と inspection report を付けられる。
- **T-DONE-8**: GLB + metadata + report + README + import notes の ZIP を、対象別 preset つきで書き出せる。
- **T-DONE-9**: 保存・再開でき、**元の 3D モデルを絶対に上書きしない**。

### 2.3 共有の完成系

- **S-DONE-1**: Home / Project Dashboard から 2D・3D いずれのプロジェクトへも入れる。
- **S-DONE-2**: 最小構成で軽く起動し、**必要な機能だけをインストール**して広げられる。使わない機能の重さを負わない。
- **S-DONE-3**: 2D と 3D が、別最適化されつつ同じ操作感で行き来できる。
- **S-DONE-4**: 失敗・制限が文章で分かり、手動で必ず回復できる。
- **S-DONE-5**: どのモジュール構成でも同じプロジェクトを開け、未知データを壊さない。

## 4. 機能仕様テンプレート（1 機能 1 ファイル）

各 `features/F-*.md` は次を埋める。

```md
# F-<領域>-<番号> <名称>

- 状態 / エディション / 所属パイプライン段
- 完成系での位置づけ（由来する D-DONE / T-DONE / S-DONE を明記＝逆算リンク）
- 対象モジュール / Core との関係（関係マップのノード ID）

## ユーザーストーリー
- 誰が・何を・なぜ

## 機能要件（FR）
- FR-1 …（番号付きで、検証可能な粒度）

## 非機能要件（NFR）
- 性能 / 信頼性（非破壊・データ非損失）/ オフライン / アクセシビリティ / 軽量

## データ・保存への影響
- schema 接点、不変条件、UI 一時状態を保存に混ぜない境界

## 依存機能
- dependsOn: F-…（先に要る機能）

## 受け入れ条件（Acceptance）
- 「こうなれば done」の観測可能な条件

## 非目標
- この機能でやらないこと

## 関係マップ linkage
- touch する node / edge（`APP_RELATIONSHIP_MAP.md` へ昇格登録）

## Open questions
- 未決事項（推測で埋めない）
```

## 5. 詳細化の推奨順（上流の土台から）

依存の土台を先に固めると、後続機能の仕様が安定する。

1. **土台**: F-CORE-01（済）→ F-CORE-05（保存形式）→ F-CORE-03（非破壊）→ F-CORE-04（Undo/Redo）→ F-CORE-09/10（Host・権限）。
2. **モジュール機構**: F-MOD-01〜04（カタログ・導入・依存・ライセンス）。install-on-demand の中核。
3. **性能・外殻**: F-PLT-01〜05。軽量前提を計測で裏づける。
4. **共通フレーム**: F-CORE-06（検品）・F-CORE-07（書き出し）・F-CORE-08（Dashboard）。
5. **2D パイプライン**: F-2D-01 → 04 → 09/10 → 07/08 → 11/12 → 15/16 の順（作る→整える→検品→書き出す）。
6. **3D パイプライン**: F-3D-01 → 03 → 05 → 06/07 → 10/11/12 → 14/15。
7. **UX 横断**: F-UX-01〜05 は各段と並走。

> 順序は目安。関係マップのエッジ（依存）と Roadmap の Stage に整合させる。ユーザーの指示で順を変えてよい。

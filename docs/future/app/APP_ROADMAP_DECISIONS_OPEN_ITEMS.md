## 1. 段階の考え方

現行の 2D/3D 計画と同じく、**壊さない順序**で概念を積む。飛ばさない。

```txt
Concept Gate（この文書一式のレビューと受理）
  ↓  ← 人間承認が必要
Stage 0：土台の確定（外殻・Core 契約・性能 budget・モジュール機構の骨格）
  ↓
Stage 1：2D Edition 最小一気通貫（作る→整える→検品→書き出す、行き止まりなし）
  ↓
Stage 2：モジュール化の実証（2D の機能をコアから外に出し、install-on-demand を成立させる）
  ↓
Stage 3：3D Edition 最小一気通貫（読み込み→検品→整える→軽量化→書き出す）
  ↓
Stage 4：拡張カタログの拡充と相互運用（VRM 等）／プロファイル整備
  ↓
Stage 5：完成判定（成功条件 6 項目を証拠つきで満たし、人間が宣言）
```

各 Stage は「終わったときの状態」で定義し、機能名の羅列で定義しない。

| Stage | 一言でいうと | 終わったときの状態 |
|---|---|---|
| Gate | 始めてよいかの確認 | 概念文書が受理され、Stage 0 の判断材料（実測すべき項目・確認すべきライセンス）が列挙されている |
| 0 | 痩せた土台 | 外殻方式・Core 契約・保存形式方針・性能 budget・モジュール manifest 骨格が決まっている（実測とライセンス確認つき） |
| 1 | 2D の一続き | 最小構成で 1 つの 2D アセットを作成〜書き出しまで迷わず完了できる |
| 2 | 本当に外せる | 主要な 2D 機能がモジュールとして入れ外しでき、未インストール分が起動・常駐・サイズに乗らないことを実測で示せる |
| 3 | 3D の一続き | GLB/glTF を読み込み・検品・整備・軽量化・書き出しまで、2D と同じ操作感で通せる |
| 4 | 広げる | VRM 等の相互運用と、目的別プロファイルが揃う。第三者拡張の可否を判断する |
| 5 | 完成 | 成功条件（Vision 8 章）を証拠つきで満たす |

## 2. Concept Gate：人間が次に判断すること（順序）

1. この `docs/future/app/` 一式のレビューと受理（draft → reviewed）。
2. 本コンセプトを、現行製品と**別系統**として維持することの承認（現行 2D Pro Gate / 3D ロードマップを妨げない前提）。
3. Stage 0 で実測・確認すべき項目（3〜4 章）に着手してよいかの判断。**着手＝実装開始ではなく、実測とライセンス調査の許可**。

> Concept Gate を通っても、それは Stage 0 の「調査・実測・設計」の許可であり、製品実装・dependency 追加・pivot の許可ではない。実装フェーズにはさらに別 Gate を置く。

## 4. Open Items（推測で埋めない未決事項）

- **保存形式の継承 vs 新設**: 現行の資産・migration 知見をどこまで持ち込むか。互換性を壊す判断は人間確認必須。
- **配布と信頼**: オンライン取得を採る場合の署名・検証・改ざん防止・完全オフライン動作の保証。
- **第三者拡張の安全性**: 開放するなら権限モデル・サンドボックス強度・審査。開放しないなら公式モジュールの拡充速度。
- **外殻・言語の体制**: Rust 等を採る場合の開発・保守体制と、OS 別 WebView 差の検証。
- **性能 budget の実数**: 冷間起動・常駐メモリ・最小インストーラの具体値と対象実機。仮値（Arch 7 章）を実測で置換する。
- **価値の写像**: 「販売しない」前提のまま、狙う価値帯（Vision 5 章）をどう体験へ落とすか。
- **AI・3D 生成の境界**: 本体非内蔵の原則を保ちつつ、外部処理との接続をどこまで許すか（現行 ADR-0017 の思想を参照）。

## 5. リスク台帳（APP-RISK）

| ID | リスク | 影響 | 予防 |
|---|---|---|---|
| APP-RISK-01 | モジュール化が過剰になり、細分化自体が複雑さ・重さを生む | 「軽くするための分割」が逆効果 | 分割は利用者の選択単位で行い、内部 API の細粒度分割と混同しない。プロファイルで束ねる |
| APP-RISK-02 | 未インストール機能がコア/UI に痕跡を残し、ゼロコスト原則が崩れる | 起動・常駐が徐々に太る | footprint 宣言 + 継続実測 + 退行検知を Gate 条件にする |
| APP-RISK-03 | 重い描画・処理依存が Core/Shell へ染み出す | 最小構成が重くなる | 重い依存は Edition/モジュール境界の内側に閉じる。Core への依存を lint/レビューで禁止 |
| APP-RISK-04 | モジュール構成差でプロジェクトが壊れる／データが失われる | 信頼の致命傷 | 「未知データ非破壊・同一 project を全構成で開ける」を不変条件として test で守る |
| APP-RISK-05 | ライセンス未確認の依存を勢いで採用 | 配布不能・法的リスク | verified=false は導入導線を出さない。採用は専用評価記録で人間承認 |
| APP-RISK-06 | 本コンセプトが現行 2D Pro Gate / 3D ロードマップの集中を乱す | 現行製品の完成が遅れる | 別系統として隔離（APP-D-01）。現行の作業指示を本コンセプトで上書きしない |
| APP-RISK-07 | 「ネイティブ化」を理由に安全境界（サンドボックス・権限）を緩める | セキュリティ低下 | capabilities 最小権限（Arch 3 章）を Core が強制。ネイティブ権限は明示・限定 |

- 確認できないことは推測せず、Open Items（4 章）と保留（APP-O-*）に置いた。
- 実装・dependency・schema・価格は確定していない。すべて概念であり、人間承認と専用 Gate を経ない限り着手しない。

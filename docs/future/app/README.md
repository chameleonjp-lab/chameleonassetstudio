## 1. 5 分で分かる要約（非エンジニア向け）

いまの Chameleon は「ブラウザの中で安全・軽量に動く」ことを優先し、そのために色々な制約を受け入れている。この構想は、その制約を外した **別物**を考える記録である。

目指すのは、2D と 3D のゲーム用アセットを「作る・整える・検品する・書き出す」ための **ネイティブアプリ**。ポイントは 3 つ。

1. **ネイティブ**: ブラウザの制限（容量・メモリ・並列処理・ファイル操作）を前提にしない。OS の力を正面から使う。
2. **目的で分ける**: 2D 制作と 3D 整備は、同じ見た目・操作感を保ちつつ、別のエディションとして最適化する。1 つの重いアプリに全部を詰めない。
3. **必要な機能だけ足す**: 最小構成で軽く起動し、利用者が自分のニーズに合わせて機能を後からインストールして広げる。**使わない機能の重さを、使う人に負わせない。**

なぜこの形か——ツールは「機能が少ないと使われない」が、「重すぎても使われない」「使わない機能で重いと離れる」。この矛盾は、機能を**カタログとして持ちつつ、使う人が使う分だけ入れる**モジュラー方式でしか解けない。それがこの構想の背骨である。

守ること: 現行の 2D/3D 作業を邪魔しない。元データを壊さない。ライセンス未確認のものを使わない。3D 生成 AI を本体に入れない。そして——**着手は禁止。これは知見の記録であり、実装許可ではない。**

- 本コンセプトは現行の `docs/future/2D_COMPLETION_ROADMAP.md`（2D Pro Gate）や `docs/future/3d/`（3D 四段階計画）を **置き換えない・妨げない**。現行製品側の作業は、これまで通り優先して進める。

- 現行 docs（`docs/`, `docs/future/`, `docs/future/3d/`, `docs/adr/`）を読み、引き継ぐ知見と引き継がない制約を仕分けた。

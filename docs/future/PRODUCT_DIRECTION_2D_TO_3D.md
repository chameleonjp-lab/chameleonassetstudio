# Product Direction: 2D Completion First, Then 3D

最終更新日: 2026-07-07  
対象リポジトリ: `chameleonjp-lab/chameleonassetstudio`  
文書種別: 2D 完成優先と 3D 展開方針  
上位文書: `docs/REQUIREMENTS_SPECIFICATION.md`, `docs/IMPLEMENTATION_PLAN.md`, `docs/future/POST_PHASE17_REQUIREMENTS.md`, `docs/future/THREE_D_ASSET_PREPARATION_REQUIREMENTS.md`

---

## 1. 目的

この文書は、Chameleon Asset Studio を今後どの順序で伸ばすかを定義する。

結論は次の通りである。

1. まず、2D ブラウザゲーム用アセット制作ツールとして完成度を最優先で高める。
2. 3D は、2D の完成後に同じ思想と見た目を持つ別画面として追加する。
3. 2D と 3D は用途が違うため、同じ編集画面には混ぜない。
4. リポジトリは当面同一のまま運用し、分離は依存・CI・bundle size・外部処理の重さが明確になった時点で検討する。

この文書は 3D 実装を開始する指示ではない。現時点の目的は、2D 完成を妨げないように将来の 3D 展開方針を固定することである。

---

## 2. 現在の最優先方針

Chameleon Asset Studio の最優先プロダクトは、2D ブラウザゲーム用アセット制作ツールである。

2D Studio では、次を破綻なく行えることを完成条件の中心に置く。

- 画像、手描き、図形、パーツを取り込む。
- キャラクター、アイテム、背景、タイル、ギミック、エフェクトを整える。
- 原点、アンカー、当たり判定、ゲーム属性を付ける。
- フレームアニメーションや簡易リグの焼き込みを扱う。
- PNG / WebP / `asset.json` / Sprite Sheet / Atlas / export ZIP を書き出す。
- `.casproj` で保存し、再開できる。
- Canvas 2D / PixiJS / Phaser などの 2D ブラウザゲームへ組み込める。

3D への関心があっても、2D Studio の完成前に 3D 都合で 2D の仕様、UI、テスト、書き出しを不安定にしてはいけない。

---

## 3. 2D Studio の役割

2D Studio は、ブラウザゲームで使う 2D アセットを作成・編集・ゲーム用データ化するための主画面である。

主な役割は次の通りである。

| 領域 | 役割 |
|---|---|
| 画像編集 | 取り込み、トリミング、透過、色調整、輪郭、軽い補正を行う |
| レイヤー / パーツ | 2D の見た目とパーツ構造を整理する |
| ゲーム用メタデータ | 原点、アンカー、当たり判定、属性、タイル設定などを付ける |
| アニメーション | フレーム、タイムライン、簡易リグの焼き込みを扱う |
| 書き出し | PNG / WebP / JSON / Atlas / ZIP / helper snippet を出す |
| 保存 | `.casproj` と IndexedDB を使って 2D 制作状態を再開可能にする |

2D Studio は、画像編集ソフトや総合ゲームエンジンではない。目的は、2D 素材をゲームに入れやすい形へ整えることである。

---

## 4. 3D Asset Preparation の役割

3D Asset Preparation は、2D Studio 完成後に追加を検討する将来画面である。

この画面の役割は、3D モデルを作ることそのものではなく、既存または外部生成された 3D モデルをゲームで使いやすい状態へ整えることである。

主な役割は次の通りである。

| 領域 | 役割 |
|---|---|
| 読み込み | GLB / glTF / OBJ などの 3D ファイルを取り込む |
| 表示 | 3D プレビュー、カメラ操作、bounds 確認を行う |
| 検品 | ファイルサイズ、頂点数、三角形数、材質数、テクスチャ、原点、スケールを確認する |
| メタデータ | 3D の origin、feet、anchors、colliders、unit、axis などを付ける |
| 軽量化 | glTF / GLB の最適化候補を確認し、必要に応じて外部処理も検討する |
| 書き出し | GLB / glTF / JSON sidecar / ZIP など、3D 用の別書き出しを扱う |

初期の 3D Asset Preparation では、画像から 3D を生成する AI を標準搭載しない。まず、読み込み、表示、検品、メタデータ付与、書き出しを検討する。

---

## 5. 2D と 3D の関係

2D Studio と 3D Asset Preparation は、同じ Chameleon Asset Studio の思想と見た目を共有する。

共有するもの:

- ローカル中心で制作できること。
- ゲームに組み込みやすいメタデータを付けること。
- 元データをできるだけ保持し、破壊的編集を避けること。
- PC / iPad / スマホで破綻しにくい操作設計を目指すこと。
- 外部サービスやクラウドを必須にしないこと。
- 見た目のトーン、用語、ナビゲーションの考え方を揃えること。

ただし、2D と 3D は用途、座標系、データ量、編集操作、書き出し先が違う。そのため、同じ編集画面に混ぜない。

禁止する混在例:

- 2D のキャンバス編集画面に 3D モデル編集パネルを無理に入れる。
- 2D の `Asset` / `Layer` / `Frame` / `Animation` の意味を 3D 用に変更する。
- 2D の export ZIP に、既存仕様を壊す形で 3D ファイルを混ぜる。
- 既存の 2D E2E を 3D 導入の都合で書き換える。
- 2D の UI 完成前に、3D 入口を主導線として目立たせる。

望ましい分離例:

- ホームまたはルートで「2D Studio」と「3D Asset Preparation」を選ぶ。
- 2D Studio と 3D Asset Preparation は別 route / 別画面 / 別 state として扱う。
- 共通 UI コンポーネントやデザイン語彙は共有してよいが、編集モデルは混ぜない。
- 3D 関連コードは lazy load や明確な module 境界を検討する。

---

## 6. 互換性を壊さないルール

3D 都合で、既存 2D のデータ形式とテストを壊してはいけない。

必ず守るルール:

1. `asset.json` の既存フィールドの意味を 3D 用に変更しない。
2. `.casproj` の既存構造と読み込み互換性を、3D 都合で壊さない。
3. export ZIP の既存ファイル名、配置、内容を、3D 都合で削除・移動・意味変更しない。
4. 既存 E2E の期待値を、3D 導入の都合で緩めない。
5. 2D の座標系、原点、アンカー、当たり判定、frame、animation の意味を変えない。
6. 3D のメタデータが必要な場合は、2D の形式へ押し込まず、3D 用の別 schema / sidecar / export を検討する。
7. 2D と 3D の共有が必要な場合も、先に docs で境界と互換性を定義してから実装する。

`asset.json` / `.casproj` / export ZIP / 既存 E2E に影響する変更は、自動判断で進めない。`docs/future/FABLELESS_DEVELOPMENT_GUIDE.md` と `docs/future/CODEX_OPUS_AUTOMATION_WORKFLOW.md` に従い、Opus 4.8 レビューまたは人間確認へ戻す。

## 7. リポジトリ方針

当面、2D Studio と将来の 3D Asset Preparation は同一リポジトリで扱う。

今すぐリポジトリ分離しない理由:

- まだ 2D 完成が最優先であり、分離作業そのものが 2D 完成を遅らせるため。
- 3D の採用ライブラリ、依存、bundle size、外部処理の重さがまだ確定していないため。
- docs、デザイン語彙、プロダクト思想、PR 運用を同じ場所で管理する方が、初期の方針ぶれを防ぎやすいため。
- 3D 実装前の段階では、分離による利点より運用コストの方が大きいため。

ただし、将来も必ず同一リポジトリに固定するという意味ではない。

---

## 8. 将来リポジトリ分離を検討する条件

次のいずれかが明確になった場合、2D と 3D のリポジトリ分離、package 分離、workspace 分離、または別アプリ化を検討する。

| 条件 | 分離を検討する理由 |
|---|---|
| 3D dependencies が重い | 2D ユーザーに不要な install / audit / build 負荷を与えるため |
| 3D により bundle size が大きくなる | 2D Studio の初期ロードやモバイル体験を悪化させるため |
| 3D CI が長い、または不安定 | 2D の通常 PR の feedback loop を遅くするため |
| WASM / native / GPU / Python など外部処理が必要 | ブラウザ 2D ツールとは運用・配布・テスト条件が変わるため |
| 3D のライセンス確認や配布条件が複雑 | 2D 本体の安全な配布判断に影響するため |
| 3D のリリース周期が 2D と大きく違う | 2D の安定版と 3D 実験版を分ける必要が出るため |
| 3D の保守担当やレビュー観点が分かれる | PR 境界と責任範囲を明確にする必要があるため |

分離を検討する場合も、最初に docs で方針を決める。コード移動、package 分割、workflow 変更、dependencies 追加は別 PR とし、2D の既存機能と export 互換性を壊さないことを完了条件にする。

---

## 9. 実装順序の原則

今後の作業順は次を原則とする。

1. 2D Studio の不足を埋める。
2. 2D の保存、読み込み、書き出し、E2E を安定させる。
3. 2D の docs と実装差分を減らす。
4. 3D は調査フェーズで dependencies、bundle size、ライセンス、CI 影響を確認する。
5. 3D を実装する場合は、2D と別画面、別 state、別 export として最小導入する。
6. 3D 導入後も、2D の既存データと既存 E2E を回帰させない。

この順序に反する提案が出た場合は、便利そうに見えても一度止め、2D 完成への影響を確認する。

---

## 10. この文書で実装しないこと

この文書は方針文書であり、次を実装しない。

- アプリ本体の UI 変更。
- GitHub Actions workflow の変更。
- dependencies の追加。
- 3D 読み込み、表示、検品、軽量化、書き出しの実装。
- `asset.json` / `.casproj` / export ZIP の仕様変更。
- 既存 E2E の変更。

> **現状:** 現在のブラウザ保存は IndexedDB と autosave queue を中心とし、`.casproj` の読み書き、保存状態、使用量表示を持つ。スマホは軽編集・確認・書き出しを優先する設計である。
> **2D 完成形の変更:** スマホでも、プロジェクトを開く、作る・直す、ゲーム用情報を確認・修正する、検査する、書き出す、バックアップするという全工程を完了できるようにする。画面構成と一括作業の効率は PC / iPad と分けてよい。

## 1. 目的

ブラウザで開けるだけでは、制作ツールとして使えるとは言えない。

Chameleon は、PC、iPad、スマホで同じ編集元を安全に扱い、端末差、ブラウザ保存の消失、容量不足、画像が重い時、通信がない時でも、現在の状態と戻り方を説明できる必要がある。

この文書は、端末ごとの操作、ローカル保存、復旧、性能、アクセシビリティ、安全性の完成条件を定義する。

## 2. 共通の約束

1. **ローカル中心**
   アカウント、クラウド同期、外部サービスを必須にしない。プロジェクトは端末のブラウザ保存と `.casproj` で扱う。
2. **同じデータ、異なる画面**
   PC の複雑な画面をスマホへ縮小コピーしない。端末に合わせて操作を組み替えるが、保存データの意味と必要な作業は共通にする。
3. **状態を隠さない**
   保存中、保存失敗、容量不足、画像欠落、未対応形式、未検証の書き出しを明確に表示する。
4. **元解像度を捨てない**
   スマホで軽量プレビューを使っても、正本の元画像と編集データを無断で縮小・置換しない。
5. **戻れることを優先する**
   自動修正、AI 提案、一括変更、変換、書き出しで元を上書きしない。操作前後の比較、Undo、バックアップ、復元を用意する。

## 3. 端末別の完成条件

| 端末 | 主な画面 | 必須の作業 | 操作上の方針 |
|---|---|---|---|
| PC | 左ツール、中央キャンバス、右プロパティ、下タイムラインを同時に表示できる。 | 全編集、複数選択、一括作業、精密な判定編集、長いアニメーション、対象別書き出し。 | マウス、キーボード、ショートカット、数値入力を併用する。 |
| iPad / タブレット | 中央キャンバスを優先し、左右・下のパネルを畳める。 | PC と同じ成果を作る、検査する、書き出す、復元する。 | タッチ、Apple Pencil、外部キーボードを使える時に活かす。筆圧などが使えない端末でも同じ結果に到達できる入力を残す。 |
| スマホ | ホーム、作る・直す、動き、ゲーム用情報、確認・書き出しを作業単位で切り替える。 | プロジェクトを開く、作成・取り込み、必要な修正、原点・anchor・collider の確認・数値修正、検査、書き出し、`.casproj` のバックアップ・復元。 | タップ対象を十分に大きくする。精密操作は拡大、ハンドル、数値入力で可能にする。大量の一括処理は隠さず PC / iPad 推奨と表示する。 |

優先して確認する実機は、iPhone 17 Pro、iPhone 11 Pro、iPad Pro 2018 とする。スマホ最小レイアウトは iPhone SE 級でも横スクロールや入力ズームで破綻しないことを確認する。Android Chrome、PC の Chrome / Edge / Firefox、iPad / iPhone Safari はリリース前の確認対象に含める。

## 4. タッチ・キーボード・画面操作

### 4.1 タッチ操作

- キャンバス内のドラッグ、パン、ピンチと、画面全体の通常スクロールを分ける。
- アプリ全体に `touch-action: none` を指定しない。キャンバスに必要な操作だけを閉じ込める。
- 長押し、テキスト選択、ブラウザの拡大などとの衝突を、画面ごとに確認する。
- 判定、anchor、timeline などの精密操作は、拡大、ハンドル、数値入力の3経路を提供する。
- 1本指と2本指の役割を説明し、意図しないパンやズームで編集値を変えない。

### 4.2 キーボードとポインター

- PC と外部キーボード付き iPad では、Undo / Redo、削除、コピー、貼り付け、保存、ズームなどの基本操作を使えるようにする。
- すべての重要操作に、マウス・タッチ以外の到達経路を残す。
- ショートカットは入力欄での文字編集と衝突させない。
- hover 専用の情報を必須にしない。タッチ端末では選択後の表示、ラベル、数値入力で同じ情報に到達できるようにする。

## 5. 保存、バックアップ、復旧

### 5.1 保存の役割分担

| 保存先 | 役割 | 注意 |
|---|---|---|
| IndexedDB | 作業中のローカル保存と autosave。 | ブラウザデータ消去、容量回収、端末故障で失われる可能性がある。唯一のバックアップではない。 |
| `.casproj` | 再編集用の可搬バックアップ、端末間移動、復元。 | 参照画像を含め、完全性を検査してから作る。 |
| export ZIP / PNG | ゲームへの配布物。 | 再編集用の正本ではない。 |

### 5.2 必須の状態表示

常に次を分かるようにする。

- 保存済み / 保存中 / 保存待ち / 保存失敗。
- 端末内保存の概算使用量と空きが不足する可能性。
- 画像 Blob、元データ、編集用データの欠落。
- `.casproj` の作成日時、読み込み時の migration、未保存変更。
- 現在の出力が `verified` か、手順のみか、未対応か。

### 5.3 将来の復旧機能

次は 2D 完成に必要な候補であり、保存構造を変える前に別設計する。

- 保存失敗後に最後に安全に保存できた版へ戻す。
- 変更前の復元点を作り、誤操作から戻れるようにする。
- ゴミ箱または論理削除を使い、削除直後に完全消去しない。
- 読み込み中断、ページ終了、クラッシュ、容量不足の後に、復旧可能な状態を明示する。
- 外部 storage を使わず、Files アプリやブラウザの共有機能から `.casproj` を保存・読み込む導線を一級機能にする。

形式移行、破壊的な一括変更、既存プロジェクトへの上書き読み込みの前には、整合が取れた復旧点を作る。復旧点の個数、画像 Blob を重複保存しない方法、削除済みプロジェクトの保持期間は、保存構造を変更する設計 PR で決める。容量不足時に唯一の復旧点を自動削除してはいけない。

### 5.4 読み込みと容量不足

- `.casproj` は一時領域で完全性を検査してから、既存プロジェクトと別の作業として確定する。
- 壊れたファイル、画像不足、古い形式、より新しい未対応形式を区別して表示する。
- migration 前の元ファイルを残せるようにし、適用した migration を表示する。
- 保存容量を取得できるブラウザでは、予防警告と `.casproj` 書き出しの導線を出す。取得できない環境では、空き容量を推測して表示しない。
- 保存に失敗した場合は、原因、保存できていない範囲、最後に整合していた保存状態、可搬バックアップを出せるかを文章で示す。

自動同期やクラウド復元を、本書だけを根拠に導入してはいけない。

## 6. 性能と大きな画像

### 6.1 現在の下限

既存要件を下限として維持する。

- 取り込み画像は1枚あたり 25MB、最大 4096 x 4096 を初期上限とする。
- 重い画像処理は Web Worker を使い、画面を不必要に止めない。
- Safari 系の画像デコードの失敗を考慮し、代替経路を維持する。
- PC / iPad の通常編集は 60fps を目標にし、スマホで長時間 30fps 未満が続かないよう確認する。

### 6.2 2D 完成形で追加する基準

性能について、実測していない数字を宣伝しない。実装前に `PERFORMANCE_BUDGET` または同等の文書を作り、次を固定する。

- 対象端末、OS、ブラウザ、テストする画像・プロジェクトの規模。
- 起動、プロジェクトを開く、画像を取り込む、編集、タイムライン再生、書き出し、復元の計測方法。
- 主スレッドを占有する処理、Web Worker に移す処理、取消可能な処理。
- メモリ警告、画像読み込み失敗、書き出し失敗時の画面と復旧方法。

スマホでは、長辺 2048px 程度の軽量プレビューを使ってよい。ただし、元解像度・編集元・最終書き出しの情報を縮小版で置き換えない。

### 6.3 性能確認の代表ケース

| ケース | 確認すること |
|---|---|
| 4096 x 4096 の透明 PNG | 取り込み、表示、縮小プレビュー、編集、書き出し、保存失敗時の説明。 |
| 多レイヤーのキャラクター | 選択、移動、反転、frame 再生、anchor / collider 表示が操作不能にならない。 |
| 複数フレームのアニメーション | タイムライン再生、onion skin、判定表示、書き出しがメモリ不足で壊れない。 |
| 多数タイル / atlas | padding / trim / multi-page 導入後の CPU、メモリ、出力時間。 |
| 復元・移行 | 大きな `.casproj` の読み込み中断、古い version、画像欠落、保存容量不足。 |

### 6.4 オフラインと更新

- 通常の作成、編集、保存、書き出しは、アカウントや常時接続を必要としない。
- 2D Pro では、一度正常に読み込んだアプリが、対象ブラウザでオフラインでも既存プロジェクトを開き、編集、保存、書き出しできることを確認する。
- これを実装する場合、アプリ資産のキャッシュ、更新中に未保存変更を失わないこと、利用者が保存後に更新を選べることを別設計で決める。
- 外部 AI、画像検索、外部 CDN が必要な操作は、オフラインの基本経路から分離して表示する。

## 7. アクセシビリティ

- 色だけで判定用途、選択状態、警告、保存状態を伝えない。線種、アイコン、ラベル、文章を併用する。
- ボタン、入力、一覧、状態表示には、読み上げ可能な名前と変化の通知を付ける。
- キーボードだけで、ホーム、プロパティ、timeline、書き出し、警告へ到達できるようにする。
- 文字サイズ、コントラスト、タップ対象、フォーカス表示を端末ごとに確認する。
- アニメーションや点滅は、必要に応じて止める / 弱める選択を検討する。

## 8. プライバシーと安全性

- 端末内で完結する操作を、許可なくネットワークへ送らない。
- 外部 AI、画像変換、クラウド保存を将来追加する場合、送信先、対象、保存期間、費用を操作前に示し、明示的に承認を取る。
- API key、アクセストークン、Cookie、秘密鍵、個人情報をプロジェクト、ZIP、ログ、共有テキストへ保存しない。
- SVG、ZIP、atlas JSON、画像、`.casproj` はすべて信頼しない入力として扱う。展開サイズ、ファイル数、相対パス、外部 URL、画像寸法を検査する。
- AI の提案は新しいレイヤーまたは派生素材として置き、元の素材を自動上書きしない。

## 9. 端末・保存・性能の品質 gate

2D 完成前に、次を確認する。

1. iPhone 17 Pro、iPhone 11 Pro、iPad Pro 2018、iPhone SE 級レイアウト、Android Chrome、PC の Chrome / Edge / Firefox で、代表プロジェクトを開く、編集、検査、書き出し、再読み込みできる。
2. PC と iPad は全編集能力、iPhone と Android のスマホは全工程への到達能力を満たす。単に画面が開くことでは合格にしない。
3. IndexedDB の消去・容量不足・画像 Blob 欠落・読み込み中断・保存失敗を、データを壊さず説明できる。
4. `.casproj` を Files アプリまたはブラウザ共有から保存・再読み込みできる。
5. 代表的な重い素材で、性能 budget の計測結果と既知の制限を残す。
6. キーボード、読み上げ、色以外の判別、タッチ操作を確認する。
7. オフラインでできる作業と、外部通信が必要な作業を区別して表示する。

実機確認は、エミュレータと Playwright の成功だけで代替しない。自動テストは回帰防止、人手確認は端末の操作感・保存・Files 連携・メモリ挙動の確認として役割を分ける。

## 10. 実装時の分割

この文書に書かれた機能は、1 PR にまとめて実装しない。

| 目的 | 分ける理由 |
|---|---|
| スマホの画面導線 | UI と E2E をまとめた1つの完成体験として検証する。 |
| 保存・復旧 | `.casproj`、IndexedDB、migration、テストへ広く影響するため別設計にする。 |
| 性能 budget と worker | 実機計測、画像処理、依存関係を確認してから進める。 |
| アクセシビリティ | UI 全体へ影響するため、対象画面と基準を決めて段階的に進める。 |
| 外部 AI / 通信 | プライバシー、費用、鍵、ライセンスを含むため、ローカル機能と別にする。 |

`2D_COMPLETION_ROADMAP.md` の順番と gate を満たす前に、3D のための重い依存関係や外部処理を 2D bundle に混ぜてはいけない。

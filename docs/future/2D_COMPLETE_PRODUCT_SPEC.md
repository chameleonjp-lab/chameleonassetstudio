## 1. 目的

Chameleon Asset Studio を、画像を取り込んで JSON を付けるだけの道具にしない。

2D 完成形では、素材を新しく作るか外部から取り込み、元を残して修正し、ゲームで必要な情報を付け、対象のゲーム制作環境へ持ち込める一式として書き出し、後日同じプロジェクトを開き直して直せるようにする。

一文で定義すると、次の通りである。

> **PC、iPad、スマホから、再編集可能な1つの編集元を使って、2Dゲーム素材を作成・修正・検品・書き出しまで完了できる、ローカル中心の制作環境。**

ここでいう素材は、画像だけではない。原点、アンカー、当たり判定、アニメーション、タイル属性、差分、対象別の書き出し設定も含む。

## 2. この完成形の位置付け

### 2.1 品質目標

「月額 800 ドル級、買い切り 4,000 ドル級」は、販売価格や課金機能の約束ではない。次の品質を求めるための基準である。

- 作品を直したとき、何を再確認・再書き出しすべきかが分かる。
- 出力先ごとに、原点、判定、切り抜き、縮尺、名前の手直しを繰り返さない。
- 保存失敗、端末差、入力ファイル不良、未対応の出力先を曖昧にせず、利用者が戻れる。
- 生成画像や外部素材が失敗しても、手動で修正して完成へ進める。

### 2.2 置き換えないもの

Chameleon は次を一度に置き換えることを目的にしない。

- Photoshop、Krita、Procreate の全作画機能。
- Spine、Rive、Live2D の専用形式や高度なランタイム全機能。
- Unity、Godot、RPG Maker のゲーム全体を作る機能。
- アカウント、クラウド同期、共同編集、課金、企業向け契約。
- 3D モデリング、3D 生成 AI、WebGPU 必須化。

代わりに、これらの道具の間で発生しやすい「ゲーム用に整える」「データを保つ」「安全に渡す」作業を、Chameleon の中心にする。

## 3. 対象利用者と終えられる仕事

| 利用者 | 完成形で迷わず終えられる仕事 |
|---|---|
| AI と一緒に小さなゲームを作る個人 | テンプレート、空キャンバス、または画像から、キャラクター・敵・UI を作り、Canvas 2D / PixiJS / Phaser で動く出力を受け取る。 |
| 既存画像や生成画像を持つ制作者 | 背景の縁、余白、色、サイズ、フレームずれを直し、元画像を残したままゲーム用素材にする。 |
| Unity、Godot、RPG Maker などへ持ち込む人 | 対象と対象バージョンを選び、必要な画像、データ、説明、検査結果をまとめて出す。 |
| 素材を量産・保守する人 | 色違い、左右向き、装備違い、解像度違いを派生素材として管理し、元の修正を安全に反映する。 |

## 4. 完成形の制作の流れ

```txt
新規作成 / 取り込み
        ↓
編集・修正・派生素材の作成
        ↓
動き・原点・アンカー・判定などのゲーム用情報
        ↓
ゲーム内に近い確認画面と修正提案
        ↓
対象別の書き出し・取り込み確認
        ↓
再編集できる .casproj を保存
```

書き出した PNG、JSON、ZIP は配布物であり、編集の正本ではない。常に `.casproj` とその中の編集元から再生成する。

## 5. 完成形を構成する機能

### 5.1 プロジェクトと素材の管理

- 1つのプロジェクトで複数の素材を管理する。
- 同じ敵の通常、色違い、左右向き、装備違いの関係を、単なる複製ではなく派生として追えるようにする。
- 元データ、編集用データ、プレビュー用の派生物、書き出し物を混同しない。
- 自動保存、明示的なバックアップ、復元、容量表示、端末間の `.casproj` 移動を扱う。

素材の親子や派生の保存形式は、`2D_ASSET_DATA_CONTRACT.md` を先に設計してから追加する。現在の `Project` / `Asset` 型へこの文書だけを理由にフィールドを追加してはいけない。

### 5.2 新規作成と取り込み

完成形では、画像取り込みを入口の1つとして扱う。

- 空キャンバス、型別テンプレート、図形、パーツ、複製から作り始められる。
- ピクセル絵、通常のラスター画像、図形・文字をゲーム素材として扱える。
- PNG / JPEG / WebP、連番画像、sprite sheet、tileset、既知の atlas bundle を取り込める方向にする。
- SVG は任意コードとして実行せず、安全な画像またはベクター情報として扱う。
- 外部制作形式は、完全に再編集できる形式、画像として取り込む形式、元ファイルを保存して参照する形式を区別する。

対応形式の確定は `2D_EXPORT_COMPATIBILITY_MATRIX.md` に従う。未実装の形式を「対応済み」と表現してはいけない。

### 5.3 編集と修正

ゲーム素材を整えるために、少なくとも次を扱う。

- ブラシ、消しゴム、塗りつぶし、選択範囲、図形、文字、変形、整列、グリッド、スナップ。
- 透明化、透明な縁の修正、トリミング、余白統一、リサイズ、パレット置換、色違い、反転、輪郭。
- 複数素材への安全な一括修正。ただし、実行前の対象確認と取り消しを必須にする。
- 元画像を破壊せず、編集の前後を比較し、Undo / Redo で戻せること。

高機能な絵画ソフトの全機能を再現するのではなく、キャラクター、アイテム、背景、タイル、ギミック、effect、UI / icon をゲームに入れられる状態へ整えることを優先する。`ui / icon` を正式な `assetType` に追加するかは、別の schema 設計で決める。

### 5.4 動きとゲーム用情報

- フレームアニメーション、オニオンスキン、可変フレーム時間、複製、反転、イベント印を扱う。
- 簡易リグは、再生時の専用形式ではなく、フレームへ焼き込める編集補助として育てる。
- 原点、アンカー、当たり判定、タイル衝突、背景ループ、パララックス、effect の発生位置を保存する。
- 将来のフレーム別判定は、アセット共通の判定を壊さず、上書き規則を明文化してから導入する。
- 多角形判定、骨、逆関節、状態遷移、変形は重要な候補だが、現在の rect / circle や既存の形式を壊さない別設計として扱う。

### 5.5 ゲーム内に近い確認と検品

Chameleon はゲームエンジンを作らない。ただし書き出す前に、素材がゲームでどう見えるかを確認できる必要がある。

- キャラクターの接地、原点、アンカー、攻撃・被弾判定を重ねて再生する。
- タイルのつながり、衝突、背景の継ぎ目、パララックス、ギミックの動き、effect の再生時間を確認する。
- 画像サイズ、透明な縁、命名、フレーム、判定、アンカー、tileSize、atlas の余白などを理由つきで検査する。
- 修正案は自動確定せず、プレビューと取り消しを可能にする。

### 5.6 書き出しと再生成

- 汎用画像、sprite sheet、atlas、タイル情報、ゲーム用 JSON、README、取り込み説明、検査結果を出す。
- 出力先ごとに、必要なファイル、座標変換、制限、対象バージョンを明示する。
- 同じ編集元と同じ書き出し設定から、意味の同じ出力を再生成できるようにする。
- Unity、Godot、RPG Maker などは、まずファイル一式と検証済みの手動取り込み手順を優先する。直接 plugin / addon が必要になるのは、手動取り込みでは重要なゲーム用情報を再現できない場合に限る。

## 6. 差別化の中心

作成物がゲームで使えるために満たす条件を、作成物ごとに持たせる。これを**アセット契約（Asset Contract）**と呼ぶ。

例として「Phaser 向けの敵キャラクター」を選んだ場合、必要な見た目、原点、動き、判定、atlas の余白、名前、出力構成が揃うまで完成扱いにしない。

この考え方により、次を実現する。

1. 画像を直したとき、影響する差分、アニメーション、書き出しを表示する。
2. 出力前に、ゲーム固有のミスを見つけ、理由と修正方法を示す。
3. 出力先、対象バージョン、警告、検証結果を記録する。
4. 生成画像や外部素材を元データとして残し、手動修正を自動上書きしない。

## 7. AI と外部サービスの位置付け

AI は正本ではなく、入力、修正候補、説明文、コード雛形の補助である。2D 完成の条件を AI 生成の成功に依存させない。

- 背景除去、フレーム分割、当たり判定候補、命名候補、色違い案は、人が確認する候補として出す。
- 外部 AI へ画像や文章を送る場合は、送信先、送る内容、保存期間を表示し、明示的な承認を取る。
- API key、認証情報、秘密鍵を `.casproj`、export ZIP、サンプル、ログへ保存しない。
- AI 出力は新しいレイヤーまたは派生素材として追加し、比較、Undo、元データへの復帰を可能にする。
- 著作権、商用利用、他作品との類似を AI が保証するとは表現しない。

## 8. 2D 完成の判定

次のすべてを満たして初めて、2D を完成扱いにできる。

1. **代表制作を通せること**
   キャラクター、タイル / 背景、UI または effect の3種類で、空から作る流れと画像を取り込んで修正する流れを通せる。
2. **ゲーム用情報が保たれること**
   原点、アンカー、判定、動き、タイル属性、差分が保存、再読み込み、反転、トリミング、atlas 化、書き出し後も意味を保つ。
3. **対象別の出力を確認できること**
   Generic Web、Canvas 2D、PixiJS、Phaser を基準出力として完成させ、Unity 2D、Godot 2D、RPG Maker MZ は対象・対象バージョンを固定した検証済み preset を持つ。詳しい基準は `2D_EXPORT_COMPATIBILITY_MATRIX.md` に従う。
4. **端末から完了できること**
   PC、iPad、スマホで、プロジェクトを開く、検査する、必要な修正をする、書き出す、バックアップする作業を終えられる。端末別の制限と確認方法は `2D_DEVICE_RELIABILITY_SPEC.md` に従う。
5. **失敗時に戻れること**
   旧 `.casproj`、画像欠落、保存容量不足、取り込み中断、未対応形式、書き出し失敗を、データを壊さずに説明・復旧できる。
6. **品質確認を通すこと**
   unit test、形式・移行テスト、描画比較、E2E、実機確認、対象別 fixture、ライセンス・安全性確認を、`2D_COMPLETION_ROADMAP.md` の品質 gate に従って通す。

## 9. 3D を始める条件

3D は削除しない。しかし、現在の Phase 22〜28 の順番では開始しない。

2D Pro Gate を通過するまで、3D は文書調査以外の実装を始めない。通過後に、3D は 2D と同じ Home / Project Dashboard を共有してもよいが、別画面、別データ形式、別の依存関係として `3D-0` から再開する。

3D のために、2D の `Asset`、`Layer`、`Frame`、`Animation`、`asset.json`、`.casproj`、export ZIP の意味を変えてはいけない。

# 実装計画詳細: Phase 0〜5

最終更新日: 2026-07-02

この文書は、Chameleon Asset Studio の初期土台からキャンバス編集までを定義する。

---

## Phase 0: 開発基盤

目的は、今後の実装を安全に進めるための土台を作ることである。この段階では編集機能を作り込まない。

作業:

- Vite + React + TypeScript のプロジェクトを作る。
- ESLint、Prettier、Vitest、Playwright を入れる。
- `src/app`、`src/core`、`src/features`、`src/renderers`、`src/workers` を作る。
- `npm run dev`、`npm run build`、`npm run test`、`npm run e2e` を用意する。
- GitHub Actions で build と test を走らせる。
- README に起動手順を書く。

完了条件:

- `npm install` が通る。
- `npm run dev` で起動する。
- `npm run build` が通る。
- `npm run test` が通る。
- README から開発開始手順が分かる。

---

## Phase 1: データ形式

目的は、UI より先に内部データ形式を固定することである。

作業:

- `Project` 型を作る。
- `Asset` 型を作る。
- `Layer` 型を作る。
- `Part` 型を作る。
- `Anchor` 型を作る。
- `Collider` 型を作る。
- `Animation` 型を作る。
- `Frame` 型を作る。
- `ExportPreset` 型を作る。
- `project.schema.json` を作る。
- `asset.schema.json` を作る。
- `animation.schema.json` を作る。
- `export.schema.json` を作る。
- `docs/DATA_FORMAT.md` を作る。

`Asset` の最低要件:

- id
- assetType
- name
- displayName
- canvasSize
- origin
- textures
- layers
- parts
- anchors
- colliders
- animations
- tags
- gameAttributes
- createdAt
- updatedAt

完了条件:

- サンプル asset JSON が schema 検証を通る。
- 壊れた JSON が検証で落ちる。
- version が入っている。
- migrate 関数の入口がある。
- テストが通る。

注意:

ズーム倍率、選択状態、開いているパネルなどは UI 状態であり、アセット本体へ入れない。

---

## Phase 2: 保存・読み込み

目的は、作業中プロジェクトを失わない状態にすることである。

作業:

- IndexedDB 保存を実装する。
- プロジェクト一覧を保存する。
- プロジェクト詳細を保存する。
- 画像 Blob を保存する。
- 自動保存キューを作る。
- 保存中、保存済み、保存失敗の状態を UI に返す。
- `.casproj` 読み込みの基盤を作る。
- `.casproj` 書き出しの基盤を作る。
- ストレージ使用量を取得する。

完了条件:

- 新規プロジェクトが保存される。
- 再読み込み後にプロジェクトが一覧へ出る。
- プロジェクトを削除できる。
- 保存失敗を UI に出せる。
- 画像 Blob が消えない。
- 保存と読み込みの往復テストが通る。

注意:

OPFS は使える場合だけ使う。最初から OPFS 前提にしない。まず IndexedDB で壊れない保存を作る。

---

## Phase 3: 最小 UI

目的は、後続機能を置ける画面の枠を作ることである。

作業:

- ホーム画面を作る。
- 新規プロジェクト作成を作る。
- プロジェクト一覧を作る。
- 編集画面の枠を作る。
- 上部バーを作る。
- 左ツールバーを作る。
- 右プロパティパネルを作る。
- 下タイムライン枠を作る。
- スマホ用の下部ナビを作る。

完了条件:

- PC 幅で基本レイアウトが崩れない。
- スマホ幅で横スクロールが出ない。
- iPad 幅でキャンバス領域が確保される。
- 新規プロジェクトから編集画面へ移動できる。

注意:

この段階では凝ったデザインをしない。先に操作の置き場所を固める。

---

## Phase 4: 画像取り込み

目的は、ゲーム素材の元になる画像を安全に扱えるようにすることである。

作業:

- PNG / JPG / WebP のファイル選択を作る。
- ドラッグ&ドロップを作る。
- `createImageBitmap` を使う処理を作る。
- 非対応時のフォールバックを作る。
- 元画像 Blob を保存する。
- 編集用画像を作る。
- サムネイルを作る。
- 画像サイズとファイルサイズ制限を実装する。

完了条件:

- PNG を読み込める。
- JPG を読み込める。
- WebP を読み込める。
- 透明 PNG の透明部分が保持される。
- 大きすぎる画像には警告を出す。
- 取り込み後にプロジェクトへ保存される。
- 再読み込み後も画像が残る。

注意:

画像を読み込んだだけで直接編集済みとして上書きしない。元画像と編集用画像を分ける。

---

## Phase 5: キャンバス編集

目的は、表示、選択、移動、ズーム、パンという編集の基本操作を作ることである。

作業:

- Canvas 2D または PixiJS でキャンバス表示を作る。
- 透明背景の市松模様を出す。
- 画像レイヤーを表示する。
- ズームを作る。
- パンを作る。
- 選択枠を作る。
- ドラッグ移動を作る。
- 数値入力で x / y を変えられるようにする。
- Undo / Redo の入口を作る。

完了条件:

- 画像をキャンバス上に表示できる。
- 選択できる。
- 移動できる。
- ズームできる。
- パンできる。
- 移動を Undo / Redo できる。
- スマホでキャンバス操作中にページ全体が不自然に動かない。
- パネルは通常通りスクロールできる。

注意:

`touch-action: none` をアプリ全体に指定しない。キャンバスだけ、現在のツールに応じて制御する。

# 実装計画詳細: Phase 6〜13

最終更新日: 2026-07-02

この文書は、画像編集から MVP 固定までの実装手順を定義する。

---

## Phase 6: 画像編集

目的は、取り込んだ画像をゲーム素材として整えることである。初期版では、完璧な画像加工ではなく、ゲームに使える最低限の補正を優先する。

作業:

- 矩形トリミングを作る。
- 単色背景の簡易透過を作る。
- 手動消しゴムを作る。
- 色相、彩度、明度の変更を作る。
- パレット置換を作る。
- 外側輪郭線追加を作る。
- 画像処理中の進捗表示を作る。
- 重い処理を Web Worker へ移せる構造にする。

完了条件:

- トリミングできる。
- 指定色に近い背景を透明化できる。
- 消しゴムで透明化できる。
- 色を変更できる。
- 輪郭線を追加できる。
- すべて Undo できる。
- 元画像を保持する。
- 処理失敗時に理由を表示する。

注意:

画像処理の品質を最初から完璧にしない。ユーザーが手動で直せる状態を優先する。

## Phase 7: レイヤーとパーツ

目的は、見た目の重なりと、ゲーム内で意味を持つ部位を分けて扱えるようにすることである。

作業:

- レイヤーパネルを作る。
- レイヤー名変更を作る。
- 表示と非表示の切り替えを作る。
- ロックを作る。
- 表示順変更を作る。
- レイヤー削除を作る。
- 複数レイヤーをまとめるパーツ作成を作る。
- パーツ名変更を作る。
- パーツ種別を設定できるようにする。
- パーツ pivot を設定できるようにする。

標準パーツ種別:

- head
- body
- arm_left
- arm_right
- leg_left
- leg_right
- weapon
- eye
- mouth
- shadow
- accessory
- other

完了条件:

- レイヤーを追加、削除、並べ替えできる。
- 表示、非表示、ロック状態が保存される。
- パーツを作れる。
- パーツに複数レイヤーを紐づけられる。
- パーツ種別を設定できる。
- パーツ pivot を設定できる。
- 再読み込み後も保持される。

注意:

レイヤーは表示の単位である。パーツはゲーム内の意味を持つ単位である。この 2 つを同じものとして扱わない。

## Phase 8: 原点、アンカー、当たり判定

目的は、画像をゲームで使えるデータへ変えることである。このフェーズが、普通の画像編集ツールとの差別化の中心になる。

作業:

- 原点ツールを作る。
- 原点をドラッグで設定できるようにする。
- 原点を数値入力で設定できるようにする。
- アンカー追加を作る。
- アンカー移動を作る。
- アンカー削除を作る。
- アンカー用途を設定できるようにする。
- 矩形当たり判定を作る。
- 円当たり判定を作る。
- 判定用途を設定できるようにする。
- 判定の表示と非表示を切り替えられるようにする。

標準アンカー候補:

- foot
- center
- head
- hand_left
- hand_right
- weapon
- projectile_spawn
- damage_effect
- shadow_center

標準判定用途:

- body
- attack
- pickup
- sensor
- custom

完了条件:

- 原点を設定できる。
- キャラクターの初期原点は下中央になる。
- アンカーを追加、移動、削除できる。
- 矩形判定を作れる。
- 円判定を作れる。
- 判定用途を設定できる。
- 原点、アンカー、判定が JSON に出る。
- 再読み込み後も保持される。

注意:

キャラクターでは、足元中央を原点の基本位置にする。原点がずれるとゲーム内の接地感が壊れるため、目視しやすいガイドを表示する。

## Phase 9: アニメーション

目的は、静止画像やパーツ状態をフレーム化し、ゲーム側で再生できるアニメーションにすることである。

作業:

- フレーム一覧を作る。
- 現在の表示状態をフレームとして保存する。
- フレーム名変更を作る。
- フレーム順変更を作る。
- フレーム複製を作る。
- フレーム削除を作る。
- アニメーション作成を作る。
- fps 設定を作る。
- loop 設定を作る。
- フレームごとの duration 設定を作る。
- 再生、停止、先頭へ戻るを作る。
- 原点と当たり判定を重ねてプレビューできるようにする。

標準アニメーション名:

- idle
- walk
- run
- jump
- fall
- attack
- damage
- dead
- win
- lose

完了条件:

- idle アニメーションを作れる。
- walk など別のアニメーションを追加できる。
- fps を変えられる。
- loop を切り替えられる。
- 再生確認できる。
- 原点と当たり判定を重ねて確認できる。
- アニメーション情報が JSON に出る。
- 再読み込み後も保持される。

注意:

初期版はフレームアニメーションを基本にする。ボーン、IK、メッシュ変形は作らない。

## Phase 10: 書き出し

目的は、作成したアセットをゲームへ組み込める形で出力することである。

作業:

- `asset.json` 書き出しを作る。
- PNG 書き出しを作る。
- WebP 書き出しを作る。
- Sprite Sheet 書き出しを作る。
- Atlas JSON 書き出しを作る。
- ZIP 書き出しを作る。
- 書き出し前の schema 検証を作る。
- 書き出し失敗時の表示を作る。

ZIP 構成:

```txt
asset-export.zip
├─ asset.json
├─ textures/
│  ├─ main.png
│  └─ main.webp
├─ atlas/
│  └─ atlas.json
├─ examples/
│  ├─ example-canvas.html
│  ├─ example-pixi.html
│  └─ example-phaser.html
└─ README.md
```

完了条件:

- 単体 PNG をダウンロードできる。
- 単体 WebP をダウンロードできる。
- asset.json をダウンロードできる。
- ZIP をダウンロードできる。
- ZIP に画像、JSON、サンプル、README が入る。
- 書き出し前に schema 検証される。
- スマホでも ZIP ダウンロードを開始できる。

注意:

保存先を直接選ぶ機能は、対応ブラウザだけの追加機能にする。基本は Blob URL のダウンロードで実装する。

完了条件:

- Canvas 2D サンプルで画像が表示される。
- PixiJS サンプルで画像が表示される。
- Phaser サンプルで画像が表示される。
- アニメーション再生例がある。
- 当たり判定のデバッグ表示例がある。

注意:

サンプルコードは完成ゲームではない。アセットを読み込む最小例にする。

## Phase 12: モバイル最適化

目的は、PC、スマホ、iPad のどれでも破綻せずに使える状態にすることである。

作業:

- スマホ縦レイアウトを調整する。
- スマホ横レイアウトを調整する。
- iPad レイアウトを調整する。
- ボトムシートを作る。
- タッチ操作を確認する。
- パネルスクロールを確認する。
- ダブルタップ誤動作を確認する。
- 入力欄フォーカス時の表示崩れを直す。

完了条件:

- スマホ幅 375px で横スクロールが出ない。
- キャンバス操作とパネルスクロールが両立する。
- iPad で主要編集ができる。
- PC で全機能が使える。
- タップ対象が小さすぎない。
- 書き出しボタンまで到達できる。

注意:

スマホでは全機能を同時表示しない。編集、プロパティ、タイムライン、書き出しを画面単位で切り替える。
